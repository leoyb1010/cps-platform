# P0-3 · 金额精度:整数分迁移技术方案(独立 PR)

> 状态:**已排期,独立 PR 执行**(2026-07-12 本轮 review 修复中经决策拆出)
> 背景:生产 Postgres 用 `Float`(双精度浮点)存储金额,应用层 decimal.js 的精算在 DB 侧浮点累加时失效;当前靠对账 ±1 元容差掩盖尾差。

## 为什么不在本轮批量修复里做

1. **规模大**:后端 48 个 `Float` 金额字段、前端 199 处金额显示、后端 55 处金额运算,前后端全覆盖 ≈ 300 处改动,还需重写全部涉及金额的测试断言与 seed 数据。
2. **单侧 Decimal 不安全(已排除)**:双 schema 共享同一份 `@prisma/client` 类型。若只把 `schema.postgres.prisma` 金额字段改 `Decimal`,而 CI/开发用 SQLite(Float→`number`)跑 typecheck,则生产 Postgres(Decimal→`Prisma.Decimal` 对象)的类型不匹配**在编译期无法暴露**——`a.payoutPending - share` 这类运算在运行时才炸。属于 typecheck 抓不到的隐藏 bug。
3. **真资损已止血**:本次真正的静默丢钱是 P0-1(`payoutPending` lost update),已用原子 `increment/decrement` 修复并加测试。P0-3 是存储层浮点尾差(存储规范性),单次运算已 round2 到分,现实漂移量(每次 ~1e-15)远小于当前 ±1 元对账容差,非即时资损。

## 目标方案:全链路整数分(分为最小单位,`Int`/`BigInt` 存储)

统一以「分」为金额最小单位,DB 存整数,类型全链路一致为 `number`(分),彻底消除浮点。

### 后端
- **两份 schema** 所有金额字段 `Float → Int`(单笔上限超 21 亿分=2100 万元时用 `BigInt`;当前金额级别 `Int` 足够,`gross` 已见 8420000 分级别数据,`Int` 上限 ~21 亿分安全)。字段:`Settlement` 九项、`Order.amount`、`Agent.payoutPending/settledTotal/deposit`、`ReserveRelease.amount/releasedAmount`、`Bundle.listPrice/finalPrice`、`Product.firstPrice/renewPrice` 等(以 `grep -n "Float" schema.prisma` 全量清点为准)。
- **money.ts**:改为整数分运算工具。`round2` 语义变为「四舍五入到分(整数)」;`sum/mul/splitProportional` 全部在整数域运算,分摊残差仍由末项吸收保证守恒。移除 decimal.js 依赖(整数运算无需)。
- **写入侧**:所有落库前 `元 → 分`(`Math.round(yuan * 100)`,或上游本就以分传入)。`increment/decrement` 天然整数安全。
- **读出侧/对外接口**:`/market`、`/portal`、`/cps`、`/youdao` 返回给前端与合作方时按各自约定转换(有道对外已是「分 Long」——见 `sign-webhook.ts:yuanToFen`,该处反而简化)。
- **对账**:`reconciliation.service` 容差收紧到 `0`(整数精确,恒等式 I 与守恒式 II/III/IV 应严格相等)。

### 前端
- **展示层收口**:199 处金额显示统一走 `format.ts` 的 `money()`,内部由「分 → 元」再格式化(`fen / 100`,`toLocaleString`)。理想情况下页面不再直接碰原始数值,改动集中在 `format.ts` + 各页 `money(x)` 调用点。
- **store/API 层**:后端返回分,前端类型标注 `分` 语义(建议 `type Fen = number` 品牌类型或注释约定),避免元/分混用。
- **输入侧**:表单金额录入(元)提交前 `× 100` 转分。

### 测试与数据
- 全部涉及金额的断言改为分(整数)。
- `seed.ts` 金额改分(当前 seed 已是分级别整数如 `8420000`,基本无需改,反而更自然)。
- 新增精度回归:高频冲账/释放/追偿后,恒等式严格相等(容差 0)。

## 迁移与上线

1. **数据迁移**:生产 Postgres 现有 `Float` 金额 → `Int`,写迁移脚本 `ROUND(col * 100)`(注意已有浮点尾差需一次性 round 落定;迁移前跑一次对账快照,迁移后比对总额守恒)。
2. **双 schema 同步**:两份 schema 字段类型保持一致(都 `Int`),CI 的 `check-schema-sync` 继续保证。
3. **灰度**:金额是全站核心,建议独立分支 + 完整回归(179 单测/e2e 全改 + 前端三类角色冒烟 + 生产只读对账比对),不与其它变更混发。
4. **回滚**:迁移脚本需可逆(`col / 100.0` 回 Float),或迁移前全库备份(见 `docs/backup-recovery.md`)。

## 过渡期缓解(本轮已保留 / 建议补)

- 对账容差保留 ±1 元(整数分落地后收紧到 0)。
- **建议补**:对账层加「浮点漂移监控」——记录每期 `reconcileDiff` 绝对值分布,若逼近容差上限即告警,量化真实漂移速度,为迁移优先级提供数据。
