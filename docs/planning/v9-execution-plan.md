# v9.1 执行方案 — 规整而非删减：三端全功能 · 清晰动线 · 落地页闭环

> **v9.0 → v9.1 修订说明**（按业务负责人评审意见修正三处，其余细化）：
>
> | # | v9.0 的问题 | v9.1 的修正 |
> |---|---|---|
> | 1 | B0「简洁模式导航 19→6」会把重点能力藏进抽屉——控制台是**后台工作台**，运营的火力不能收纳 | 改为**分组折叠制**：一项不减、一项不藏，主类目可折叠但永远在侧栏可见可展开（§2.1） |
> | 2 | 方案视角偏平台侧，没有系统回答「品牌方、服务商在我们五大业务上是否**全功能且清晰**」 | 新增核心章节：**三端 × 五业务能力矩阵**（§1），逐格盘点现状与缺口，并把合约/置换以「**资源平台**」形态开放进品牌门户（§2.2） |
> | 3 | C 端只有超市货架，投放到用户手上的「最后一米」缺失 | 新增**落地页生成器**（§3）：移动优先独立页 + 广告位嵌入 + 代理归因推广页，把 CPS 归因闭环缝上 |
>
> **瘦身原则重新定义（全篇遵守）**：瘦身 ≠ 减功能 / 减能力 / 删内容。瘦身 = **合理分类 + 合理布局 + 合理动线**。任何"简化"手段只允许作用于*摆放方式*（分组/折叠/密度/顺序），不允许作用于*能力本身*。
>
> 沿用 v9.0 的铁律作为第二原则：操作要小白友好——但实现路径是「动线引导 + 术语解释 + 影响预览」，不是「藏功能」。

## 一、三端 × 五业务能力矩阵（v9.1 的思考底盘）

> **原因**：这个平台的本质是「一个撮合底座 + 五种业务玩法 + 三种参与角色」。每个业务必须回答三个问题：平台运营怎么管？品牌怎么用？服务商（代理）怎么接活？任何一格是空的或含糊的，业务闭环就漏气。下表 ✅=已有且够用，🟡=有但弱，❌=缺失。

### 1.1 全景矩阵

| 业务 | 平台控制台（管） | 品牌门户（供给侧） | 代理门户（履约侧） | C 端 |
|---|---|---|---|---|
| **① CPS 自营投流** | ✅ 选品市场配置 · 代理管理 · 订单归因 · 号池 | 🟡 只能看订单/结算，**看不到"我的商品被谁在投、投得怎样"**（缺投放透视） | ✅ 选品市场领链接 · 投放计划 · 分润提现 | ❌ 落地页缺失（见 §3） |
| **② 增长合约** | ✅ 全量台账 · 审核 · 状态推进 | ✅ 发单（proposeContract）· 跟踪履约 | ✅ 接单（claimContract）· 履约进度 | — |
| **③ 资源置换** | ✅ 台账 · 仲裁 | ✅ 发起/响应置换 —— 但**埋在导航第 9 项**，且看不到"市场上有哪些可置换资源"（缺资源发现） | ❌ 暂不开放（置换是品牌间业务；预留"代理作为资源方"为 v10 议题，不在本轮） | — |
| **④ AIGC 素材** | ✅ 素材实验台 · LTV 排名 | ✅ PortalAigc | ✅ PortalAigc | — |
| **⑤ 商品 / 订阅超市** | ✅ 商品审核 · 组合规则 · 套餐受理拆单 | ✅ 上架商品 · 🟡 套餐落地页（现状只是深链生成，非真落地页） | 🟡 只在受理时被动当"归因对象"，**不能主动生成带自己归因的推广页** | 🟡 只有货架（`/market`），无"我的订阅"、无移动落地形态 |

### 1.2 从矩阵推导的缺口执行清单（本轮要补的格子）

| # | 缺口 | 属于谁 | 解法 | 批次 |
|---|---|---|---|---|
| G1 | 品牌看不到自己商品的投放透视（哪些代理在投/素材/转化/退款） | 品牌 · 业务① | 品牌门户新增「投放透视」页：按商品聚合代理投放数据（复用 claims/orders 数据，脱敏代理敏感字段——只给代理编号与量级，不给对方客户资产细节） | B2 |
| G2 | 合约/置换在品牌门户是零散两项，没有"平台开放业务"的整体心智 | 品牌 · 业务②③ | 门户导航新增「**资源平台**」分组（§2.2），并加**资源广场**页：可接合约挂单 + 可置换资源卡片流（跨品牌可见的脱敏摘要） | B1 |
| G3 | 代理的"接活"入口分裂：选品市场领链接在一处、合约接单在另一处 | 代理 · 业务①② | 门户导航新增「**接单大厅**」分组，合约挂单与选品市场并列呈现，同一套"接单"心智（§2.3） | B1 |
| G4 | C 端落地页缺失，投放最后一米断裂、归因靠运营手选 | C 端 · 业务①⑤ | **落地页生成器**（§3）：品牌官方页 + 代理归因推广页 + 广告位嵌入 | B1-B2 |
| G5 | C 端支付后动线断头（无处查开通进度） | C 端 · 业务⑤ | 「我的订阅」`/market/me`（沿 v9.0 方案，§3.4） | B2 |
| G6 | 置换资源不可发现（只能凭空对指定品牌发起） | 品牌 · 业务③ | 资源广场内「可置换资源」卡片流：品牌自愿挂出可置换资源（曝光位/会员权益/联名额度），他牌浏览并发起 | B2 |
| G7 | AIGC 埋在两个门户的导航尾部，与投放动线脱节 | 品牌+代理 · 业务④ | AIGC 独立分组（用户明确要求）；且在选品市场/投放计划页内嵌"为此商品生成素材"直达入口——素材应该长在投放动线里，而不只是一个独立页 | B1 |

> **想法**：G1-G3 的共同本质是——门户此前是"品牌/代理查自己数据的地方"，v9.1 把它升级为"**在平台上做生意的地方**"。查数据是 dashboard 心智，做生意是 marketplace 心智，后者才配得上"联运平台"四个字。

## 二、导航规整（一项不减，重新摆放）

### 2.1 控制台：分组折叠制 🔴 B1

- **原因**：19 项对新人有压迫感，但每一项都是运营火力——收纳进抽屉等于告诉用户"这些不重要"，错。正确做法是**保持全部可见性的前提下控制视觉密度**。
- **想法**：现有 6 个分组（概览 / CPS 自营投流 / 交易模式 / 接入与资金 / 风控合规 / 系统）已经分得不错，缺的是**折叠交互**：
  - 分组标题可点击折叠/展开，状态持久化（`prefs.ts` 新增 `navCollapsed: string[]`）
  - **当前路由所在分组强制展开**（防止"我在哪"迷失）
  - 折叠态显示徽标聚合（如风控合规折叠时组标题右侧仍显示红点 3）——折叠不等于失去感知
  - 新装账号默认全展开；用户折叠是主动学习成本管理，系统不替用户做决定
- **可选增强（B2）**：分组标题旁"📌 常用"——用户可把任意项 pin 到顶部"常用"区（≤5 项，prefs 持久化）。这是**用户自己做的减法**，产品永远不替他删
- **落点**：`nav.ts` 结构不动；`AppLayout.tsx` Sidebar 加折叠交互 + 徽标聚合；`prefs.ts` 加 navCollapsed/pinned
- **验收**：19 项在任何状态下都可两次点击内到达；折叠状态刷新保持；当前组永不折叠；徽标聚合正确

```
┌ 侧栏（示意）──────────────┐
│ 📌 常用（用户自定义，可空）  │
│    经营总览 · 清结算        │
│ ─────────────────────────  │
│ ▾ CPS 自营投流             │   ← 当前路由在此组，强制展开
│    选品市场 / 代理商        │
│    订单·订阅 / 数据·归因    │
│ ▸ 交易模式                 │   ← 折叠，点击展开
│ ▸ 接入与资金               │
│ ▸ 风控合规 ●3              │   ← 折叠仍见告警徽标
│ ▸ 系统                     │
└───────────────────────────┘
```

### 2.2 品牌门户：11 项平铺 → 4 组（含「资源平台」）🔴 B1

- **原因**：品牌门户现在是 11 项无分组平铺（`portalNav.ts`），"我的入驻"和"资源置换"平级排队——品牌方分不清哪些是**经营自己**、哪些是**参与平台业务**。用户点题：合约/置换是平台化业务，应该让品牌看到一个"资源平台"的整体。
- **想法**：

| 分组 | 项 | 心智 |
|---|---|---|
| **我的经营** | 经营总览 · 我的订单 · 我的结算单 · 我的工单 | 看自己的生意 |
| **商品与投放** | 我的商品 · 落地页工坊（§3 升级）· 投放透视（G1 新增） | 供给与转化 |
| **资源平台** 🆕 | 资源广场（G2 新增）· 我的增长合约 · 资源置换 | 参与平台开放业务 |
| **AIGC 创作** 🆕 | AIGC 素材 | 独立分类（用户点名） |
| **接入与开发** | 我的入驻（接入进度轨道挂此）· 开发者中心 | 一次性配置区 |

- **资源广场页（新）**：品牌视角的"平台业务橱窗"——左 tab 可接合约挂单（他牌发的 open 合约，含结算模型/目标 GMV/分成，一键去响应），右 tab 可置换资源卡片流（G6）。**这是"平台释放业务给品牌"的具象化**，也是资源平台分组的首页。
- **落点**：`portalNav.ts` 加分组结构（`PortalNavGroup[]`）；`ClientLayout.tsx` 侧栏渲染分组（复用 2.1 折叠交互）；新页 `src/pages/portal/BrandPlaza.tsx`（资源广场）+ `BrandInsights.tsx`（投放透视）；服务端 portal.controller 加两个 scoped 读端点（可接合约挂单已有 contracts open 查询可复用；置换资源挂牌需 BarterListing 轻表）
- **验收**：品牌账号登录 10 秒内能说出"我能在这个平台上做哪几类生意"（新人测试）；资源广场每张卡都有明确 CTA（去接单/去发起）

### 2.3 代理门户：8 项平铺 → 3 组（含「接单大厅」）🔴 B1

| 分组 | 项 | 心智 |
|---|---|---|
| **我的经营** | 我的投放（首页）· 我的分润 · 我的信用分 · 相关工单 | 看钱与健康度 |
| **接单大厅** 🆕 | 选品市场（CPS 领链接）· 增长合约接单 · 推广页工坊（§3，带归因落地页） | 找活干 · 一个心智 |
| **投放工具** | 我的投放计划 · AIGC 素材 | 干活的工具 |

- **想法**：服务商的核心动线是「找活 → 干活 → 收钱」。现状选品市场和合约接单分居两处，是两种"活"却要两种心智——合并进"接单大厅"后，CPS 商品（自助领链接型）与增长合约（挂单竞标型）并列为两种**订单类型**，未来新增业务型态（比如置换履约方）继续往大厅里加，扩展模型清晰。
- **落点**：`portalNav.ts` 分组；AgentMarket 与 AgentContracts 页顶部互链 tab（保持独立路由，视觉归组）；推广页工坊见 §3
- **验收**：代理账号从登录到"领到一个新链接/接到一张合约"≤3 次点击

## 三、C 端落地页生成器（本轮最重要的新能力）🔴

> **原因**：投放业务的完整链条是 `素材 → 广告位 → 落地页 → 支付 → 归因分润`。平台现在有素材（AIGC）、有支付（超市）、有分润（清结算），**唯独没有落地页**——现状 BrandLanding 只是生成 `#/market?pre=<ids>` 深链，打开还是完整超市货架，不是可投放的页面。这一米不补，"CPS 投流平台"名不副实。
>
> **想法**：落地页不是一个页面，是一个**可配置、可归因、可嵌入的转化单元**。一套渲染器，三种用法。

### 3.1 落地页形态设计

```
路由：/#/land/:pageId        （独立于超市，无导航壳，移动优先）
┌─────────────────────────┐
│  [品牌 Logo]  官方认证 ✓  │   ← 白标：用品牌主色（brandLogos 色板已有）
│                           │
│   词典 VIP + 喜马拉雅      │   ← 标题/副标题可配置
│   组合价 ¥39.9  立省 15%   │   ← 价格服务端权威（/market/quote）
│                           │
│   ✓ 查词无广告  ✓ 有声书   │   ← 卖点三条（默认取商品 equity，可改）
│                           │
│  ┌─────────────────────┐  │
│  │   立即订阅 ¥39.9 →   │  │   ← 唯一 CTA，直达支付（复用超市成套+支付流程）
│  └─────────────────────┘  │
│   平台担保 · 随时可退      │   ← 合规元素强制：扣费告知 + 退订入口（风控事前防线要求，
│   自动续费说明 · 退订入口   │      与 Risk 页"落地页强制合规元素"呼应——终于落地了）
└─────────────────────────┘
```

- **移动优先**：单列、大按钮、无侧栏；桌面打开居中卡片化。已有响应式令牌体系直接用
- **嵌入形态**：同一路由加 `?embed=1` 去掉页眉页脚 → 可 iframe 进广告位/公众号文章底部；生成器同时给出 iframe 代码片段与二维码（二维码用纯 SVG 自绘或极小依赖，评估后定）
- **合规强制**：连续包月商品的落地页**强制渲染**"自动续费价格与周期 + 退订路径"模块，不可配置关闭——这是把风控中心写着的"落地页强制合规元素（事前防线）"从口号变成产品事实

### 3.2 归因闭环（这是和"链接"最大的区别）

```
代理在推广页工坊生成 → pageId 绑定 { productIds, agentId, channel }
   ↓ 投放（信息流/公众号/私域）
用户打开 /land/:pageId → 下单 → Bundle 落库时带 refPageId
   ↓
运营受理拆单：归因代理自动预填（来自 pageId 绑定）——不再人工猜
   ↓
命中该代理在投合约 → 累加履约 GMV → 分润 → 清结算
```

- **想法**：现状套餐受理时"归因代理"靠运营手选（fulfillBundle 的 agentId 参数），等于归因靠人脑。落地页把归因参数**固化在投放物料里**，受理时自动带出，人只做确认。品牌官方落地页 pageId 不绑代理（归因为品牌自营）。
- **防作弊边界**：pageId 只做归因**建议**，运营受理时可改（改动写审计）；同一用户多 pageId 触达按"首触/末触"归因走 attributionConfig 现有配置——归因模型是已有能力，落地页只是给它喂上了真实的触点数据。

### 3.3 生成器（两个门户各一个工坊，同一套底层）

| | 品牌门户「落地页工坊」（升级现有 BrandLanding） | 代理门户「推广页工坊」（新） |
|---|---|---|
| 选品范围 | 自己 live 的商品自由组合 | 选品市场内**该品牌开放代理推广**的商品（brand 商品配置加 `allowAgentLanding` 开关，默认开） |
| 归因 | 品牌自营 | 自动绑定本代理 + 可选渠道标记 |
| 定制项 | 标题/副标题/卖点三条/主题色（白标） | 标题/卖点（价格与合规模块不可动） |
| 产出 | 链接 + 二维码 + iframe 片段 | 链接 + 二维码 + iframe 片段 |
| 台账 | 我的落地页列表：曝光(占位)/下单数/转化金额 | 同左，并联动"我的投放计划"（一个落地页=一条 claim 的延伸） |

- **落点**：
  - 新表 `LandingPage { id, brandId, agentId?, productIds, title, subtitle, points, theme, channel, status, createdAt }`（演示态入 store）
  - 渲染页 `src/pages/market/LandingPage.tsx`（复用超市的算价/支付组件，剥壳）
  - 工坊：`BrandLanding.tsx` 升级 + `src/pages/portal/AgentLanding.tsx` 新增
  - 服务端：`market.controller` 加 `GET /market/landing/:id`（公开、脱敏）；portal.controller 加 CRUD（scoped）；`bundles` 落库带 `refPageId`；`fulfillBundle` 归因预填
- **验收**：手机竖屏打开落地页 → 支付成功 ≤3 步；代理页下单后受理界面归因自动预填正确（e2e）；连续包月落地页 100% 渲染合规模块；iframe 嵌入在 375px 宽度下无横向滚动

### 3.4 「我的订阅」`/market/me`（沿 v9.0，归入 C 端动线）🟠 B2

套餐号+手机尾号免登录查询开通进度（拆单逐笔状态）、续费日、三步合规退订（原因→挽留报价→确认）。落地页支付成功页直接给进度页链接。

## 四、动线设计（保留 v9.0 精华 + 补 B 端两条）

> v9.0 的结算工作台/处置舱/行动队列/⌘K 动作化 获认可，全部保留并细化；本节新增两条 B 端动线。

### 4.1 结算日动线：结算工作台 `/settlement/run` 🔴 B1

五步 Checklist 一屏走完（拉取对账 → 处理差异[内嵌解释器] → 发起结算 → 释放准备金 → 审批提现），步骤状态持久化可中断续跑，完成生成本期结算报告写审计。**本质是现有 API 的编排，零新后端。**
- 落点：新页 `workspaces/SettlementRun.tsx`；复用 `bizApi.reconcile/clear/releaseDue/settleAgent`；清结算页顶部加「开始本期结算 →」主按钮（原页全部保留，工作台是快捷动线不是替代）
- 验收：全流程 <10 分钟；中断刷新续跑；每步审计留痕

### 4.2 风险处置动线：处置舱 `/risk/incident/:id` 🔴 B1

全屏路由容器：左=事件上下文（商户号 30 天曲线/关联订单/代理画像），中=动作带**影响预演**（"熔断 M-BL-01 → 约 ¥42万/日 按权重改道 M-MG-02"，号池权重真实计算），右=留痕时间线；完成后生成复盘卡（处置前后指标对比）进事件流。
- 落点：通用容器 `IncidentRoom.tsx`（后续结算差异/合约违约复用同壳）+ `RiskIncident.tsx`
- 验收：待办一键开舱、处置、返回原位滚动保持；预演数字与号池口径一致（单测）

### 4.3 行动队列（待办四态制度化）🔴 B1

待处理/已认领/已延后/已完成 四态 + 认领人可见 + 完成写审计；总览待办、处置舱、结算工作台统一从队列取数。
- 落点：`store.ts` 加 actions 集合；真实模式 B2 接 ActionItem 表

### 4.4 ⌘K 动作化 🟠 B1

命令面板加动作注册表（首批 10 个：退款/熔断/开始结算/切主题/邀请成员…），有权限才出现，资金类走标准 Confirm。面板底部灰字轮播示例做渐进引导。

### 4.5 品牌发单动线（新）🟠 B2

```
资源广场看同行案例 → 发起增长合约（Wizard：结算模型→目标→分成→风控条款，每步右侧实时预览"代理视角看到的挂单"）
→ 挂单进入代理接单大厅 → 有代理接单推送通知（SSE）→ 品牌确认 → 履约进度条
```
- **想法**："代理视角实时预览"是关键小心思——发单方最常犯的错是条款写得只有自己懂；让他边写边看对手盘视角，挂单质量自然上去。
- 落点：品牌门户 proposeContract 已有，升级为 Wizard + 预览面板；通知复用 portal notifications + SSE

### 4.6 代理接单动线（新）🟠 B2

```
接单大厅（挂单卡：品牌/模型/目标GMV/分成/风控责任 一屏比价）→ 详情页"接单模拟器"
（拖动预估 GMV 滑杆 → 实时算预估分润/预估准备金占用/违约风险提示）→ 接单 → 进入我的接单履约看板
```
- **想法**：接单模拟器把"这单划不划算"从心算变成滑杆——服务商决策成本降一个量级，也是平台专业感的展示位。纯前端计算（结算模型公式已有）。

### 4.7 C 端购买动线（§3 落地页 + §3.4 我的订阅串起来）🟠 B2

`广告位/分享 → 落地页(3 步支付) → 成功页 → 我的订阅(进度/退订)`——全程手机可完成。

## 五、页面设计升级（Design v3，全部保留 v9.0 并细化）

### 5.1 简洁/专家模式重新定位（响应"控制台不简化"）

- **修正**：简洁/专家**不再控制任何功能的可见性**（v9.0 的导航裁剪废弃），只控制三件事：
  1. **表格列密度**：简洁 5 列核心列 / 专家全列（列定义 `simpleCols` 契约保留，但性质是"默认列视图"，简洁模式下也有"列设置"可自己加回全列）
  2. **解释密度**：简洁模式术语自动带 `<Term>` 下划线释义、危险按钮带一行后果说明；专家模式收起解释保持信息密度
  3. **总览编排**：简洁 = 「今天三件事」行动区置顶 + 全量看板在下（**不删看板**，v9.0 的"替换"改为"置顶"）；专家 = 现状
- **原因**：可见性属于权限与角色的领域（RBAC 已管），不该被"视图偏好"僭越。视图偏好只该管密度与解释量。

### 5.2 三栏工作台范式（订单页首发）🔴 B1

列表(筛选/搜索/游标分页——后端早支持前端欠账补上) | 详情(状态机时间线+关联工单/订阅) | 动作面板；<lg 折叠两级；"我的视图"保存筛选组合。工单页、代理页 B2 跟进。

### 5.3 可视化 2.0 🟠 B1-B2

- **分润 Sankey**（结算页"资金流向"视图）：GMV → 品牌留存/平台费/代理分润/准备金/冲账 五股，宽度=金额，差异标红；渲染前断言五股和===gross
- **号池热力图**：商户号×日期投诉率矩阵；**接单大厅比价卡**（4.6）；**雷达图**（代理成长面板四维）
- 图表交互三件套：区间刷选联动 KPI / 上期对比虚线 / 点击下钻明细

### 5.4 页面重造清单 🔴 B2

| 风控中心 | 去静态贴图：规则真 CRUD+命中计数（事件驱动累加），拦截漏斗由事件真值聚合，零冻结常数 |
