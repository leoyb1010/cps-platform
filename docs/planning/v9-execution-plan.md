# v9 执行方案 — 升级与优化（动线化 · 实时化 · 智能化，但小白永远能用）

> **本文档是执行级方案**：每一项都写清 ①原因（为什么值得做）②想法（设计思路）③小白形态（它在简洁模式长什么样）④落点（改哪些文件）⑤验收标准。
>
> **与 v8 的关系**：v8 解决了正确性（资金不双花、权限不越界、明暗主题、演示三入口全通）。v9 解决「顺手」——但有一条铁律凌驾于所有升级之上：
>
> ### ⚖️ 铁律：功能可以专业，操作必须小白
> 平台已有控制台 19 页 + 双门户 19 页 + 106 个 API。**每加一个功能都在偷走一点易用性**——企业软件不是死于功能少，是死于新人第一天打开时的绝望。因此 v9 的第一个工作流不是加法，是减法；且此后每个功能合入前都要过「小白测试」（见 0.3）。

## 〇、复杂度治理框架（先立规矩，再谈升级）

### 0.1 原因：为什么现在必须立这个规矩

- **现状信号**：导航 19 项、订单表 8 列、清结算页塞了分润瀑布+冲账+冻结+对账+提现五件事、术语密度高（"逆向冲账""准备金分期释放""R-NSC"）。对财务/风控老手是效率，对新人是墙。
- **已有的好底子**：系统里其实早就埋了简化机制——`简洁/专家` 双视图（`src/lib/prefs.ts` 的 viewMode）、每页指引抽屉（`GuideDrawer` + `src/lib/guides.ts`）、术语解释组件（`src/components/ui/Term.tsx`）、分步向导（`Wizard.tsx`）。**问题是它们没有被当成制度**：viewMode 只有总览页认真用了，别的页面简洁≈专家。
- **想法**：不发明新机制，把已有四件套（双视图/指引/术语/向导）升格为**全站契约**，用"复杂度预算"强制执行。

### 0.2 三层用户模型（所有设计决策的坐标系)

| 层 | 是谁 | 需要什么 | 对应机制 |
|---|---|---|---|
| **L0 小白** | 第一周的新运营、老板、演示观众 | 打开就知道"今天该干嘛"，跟着按钮走不出错 | 简洁模式（默认）+ 待办驱动 + 向导 |
| **L1 熟手** | 用了一个月的运营 | 快速定位、批量处理、少点几次 | 专家模式 + ⌘K + 筛选视图 |
| **L2 专家** | 财务/风控专业岗 | 全字段、全口径、可审计可导出 | 专家模式 + 密度切换 + 导出 |

**规矩**：新功能必须声明它服务哪一层；服务 L1/L2 的功能，**默认在简洁模式不出现或折叠**。简洁模式是产品的脸，专家模式才是抽屉。

### 0.3 小白测试（每个 PR 的合入门槛，写进 PR 模板）

1. 新人首次登录，**3 分钟内**能说出"我今天该处理什么"？（待办驱动）
2. 每个页面顶部一句话能讲清"这页是干嘛的"？（PageHeader desc 已有，保持）
3. 危险动作（钱/状态）有 **影响预览 → 确认 → 完成回执** 三段式？
4. 走到任何一步，界面上**永远有一个显眼的"下一步"**？（不许出现"然后呢？"时刻）
5. 专业术语首次出现有 `<Term>` 解释？新增术语同步进术语字典？

### 0.4 复杂度预算（量化，不靠自觉）

- 简洁模式导航 ≤ **7 项**；任何表格简洁模式 ≤ **5 列**；任何表单首屏必填项 ≤ **4 个**（其余折叠进"高级选项"）
- 新页面 = 必须有一个旧页面合并或降级为专家模式，才准上导航（**一进一出**）
- 度量落地：`flow.*` 埋点（见 §6）统计"求助率"（打开指引次数/会话）与"首任务完成时间"，连续两批恶化则冻结新功能

## 一、B0 · 先做减法（1 周量级，纯前端，立刻让小白受益）

> **原因**：加法谁都会，减法才防止"功能健全但没人会用"。B0 不加任何新能力，只把已有能力重新摆放。

### 1.1 简洁模式导航瘦身：19 项 → 6 项

- **想法**：简洁模式下侧边栏只留 `总览 / 订单 / 清结算 / 风控合规 / 品牌 / 代理`，其余（选品市场、合约、置换、AIGC、商品、超市、号池、归因、成员、审计、配置）收进底部"全部功能"抽屉（一格网格弹层，带一句话说明）。专家模式维持现状 19 项。
- **为什么这 6 个**：对应新运营的真实高频闭环——看盘 → 看单 → 结钱 → 处理投诉 → 管两类客户。号池/归因是专业岗工具，配置/成员是一次性设置。
- **小白形态**：即默认形态。切到专家模式的入口就在抽屉里，发现路径自然。
- **落点**：`src/components/layout/nav.ts` 加 `simpleNav: boolean` 标记；`AppLayout.tsx` Sidebar 按 viewMode 过滤 + 新增 `AllFeaturesDrawer`；不改路由（直链仍可达，权限守卫照常）。
- **验收**：简洁模式首屏导航 6 项；抽屉内每项有 icon + 一句话；Playwright 用例覆盖两种模式的导航渲染。

### 1.2 总览页简洁模式改为「今天三件事」

- **原因**：现在简洁模式的总览仍是 KPI 陈列——数字对小白没有行动含义。小白要的不是仪表盘，是**任务清单**。
- **想法**：简洁模式总览 = 一张大卡：`今天需要你处理 N 件事`，按严重度排前三（逾期工单 → 熔断商户号 → 待结算），每件带一个直达按钮；下面只留 3 个大数字（今日流水/净收入/投诉率）+ 一句话解读。专家模式保持现有全量看板不动。
- **小白形态**：即此形态本身。做完三件事出现"今日已清空 🎉 + 明日预告"。
- **落点**：`src/pages/Dashboard.tsx` 已有 simple/expert 分支与待办数据源（`selectTodos`），重排简洁分支即可；文案解读用规则模板（为 §4.1 异动播报预留同一插槽）。
- **验收**：新账号进入 3 分钟内可完成一件待办（Playwright 模拟）；简洁总览首屏元素 ≤ 8 个块。

### 1.3 表格减列与密度契约

- **原因**：订单表 8 列（订单号/时间/品牌/套餐/代理/通道/商户号/类型/金额）在简洁模式全量呈现，信息过载。
- **想法**：每张表声明 `simpleCols`（订单：品牌套餐/类型/金额/时间/状态 5 列）；专家模式全列 + 新增"舒适/紧凑"密度切换（行高 44→36px，令牌 `--row-h` 控制）。
- **落点**：`TableShell/Th/Td`（`primitives.tsx`）加列可见性上下文；`index.css` 加 `--row-h` 令牌；各表逐页标注（订单/结算/工单/代理先做）。
- **验收**：简洁模式四张核心表 ≤5 列；密度切换持久化（prefs.ts）。

### 1.4 术语与文案人话化审计

- **原因**：小白卡住的第一现场是名词。"逆向冲账"没有解释时是黑话，有解释时是教育。
- **想法**：把 `src/lib/terms.ts` 升格为**全站术语字典**（每条：术语/一句话人话/一个例子），扫全站给 20 个高频术语补 `<Term>` 包裹；同时做一轮**按钮文案审计**——动词开头、说结果不说机制（"退款并冲账"→ 简洁模式显示"退款（自动追回分成）"）。
- **落点**：`terms.ts` 扩充 + grep 审计清单落 `docs/planning/copy-audit.md`；改动散布各页但都是文案级。
- **验收**：20 个术语全站覆盖率 100%；新人测试（找一个没用过的人）能复述 5 个核心概念。

### 1.5 首次进入 3 步 Coach Marks

- **原因**：GuideDrawer 是"被动求助"（要自己点 ?），小白不知道有它。
- **想法**：每类入口首次进入播 3 步聚光引导（控制台：待办在这→搜索⌘K→切专家看全量；门户/超市各 3 步），跳过即永不再弹（prefs 持久化），reduced-motion 直接跳过。**只做 3 步，坚决不做 10 步长教程**——教程越长越没人看。
- **落点**：新组件 `src/components/layout/CoachMarks.tsx`（定位描 spotlight + 文案气泡）；`prefs.ts` 加 `coachDone` 键。
- **验收**：三入口各 3 步可完整播放/跳过/不复弹；对 GuideDrawer 的打开率做前后对比埋点。

## 二、B1 · 动线与骨架（2-3 周量级，v9 主菜）

### 2.1 结算工作台（`/settlement/run`）🔴

- **原因**：月结是这个平台存在的意义，但现在结算日要在清结算/对账/准备金/提现四处横跳，中断即断片。**这是全平台价值密度最高的一条动线。**
- **想法**：五步 Checklist 一屏走完，本质是**现有 API 的编排**（零新后端）：

```
┌ 结算工作台 · 2026-07 上半月 ──────────── 进度 2/5 ┐
│ ✓ ① 拉取对账      已平 6 · 差异 2                  │
│ ▶ ② 处理差异      S-2405-MG ¥31,200 [展开归因]     │
│      └ 构成：3 笔跨期退款 ¥28,400 + 通道费差 ¥2,800 │
│      └ [核销] [挂起并留言]                          │
│ ○ ③ 发起结算      待结算 2 笔 ¥168.3 万             │
│ ○ ④ 释放准备金    到期 3 笔 ¥53.9 万                │
│ ○ ⑤ 审批提现      代理申请 2 单                     │
└──────────────── 完成后生成本期结算报告 ────────────┘
```

- **小白形态**：**这个工作台本身就是给小白的**——它把专家知识（先对账再结算再释放的顺序、每步的检查点）固化成了流程。简洁模式清结算页顶部放大按钮「开始本期结算 →」直接进工作台；专家仍可在原页逐个操作。
- **步骤状态持久化**：localStorage（演示）/ Config 表（真实），中断可续；每步完成写审计。
- **落点**：新页 `src/pages/workspaces/SettlementRun.tsx`；复用 `bizApi.reconcile/clear/releaseDue/settleAgent`；报告生成复用 CSV 导出 + 审计写入；路由挂 `settlement.clear` 权限。
- **验收**：全流程 <10 分钟（Playwright 计时）；中断刷新后从第 N 步继续；每步审计留痕齐全。

### 2.2 风险处置舱（`/risk/incident/:id`）🔴

- **原因**：处置一个熔断要自己去号池页找曲线、订单页找异常单、代理页看画像——上下文全靠人肉聚合。
- **想法**：全屏聚焦容器（不是弹窗，是路由，可分享链接可回退）：左=事件上下文（商户号 30 天投诉曲线、关联异常订单、同代理近 7 天记录），中=处置动作带**影响预演**（"熔断 M-BL-01 → 约 ¥42万/日 流水将按权重改道 M-MG-02"，数据从号池权重真实算），右=留痕时间线。处置完成 → 48h 后（演示态立即）生成复盘卡进事件流。
- **小白形态**：影响预演就是给小白的安全网——**看得见后果才敢按按钮**。所有动作沿用 Confirm 三段式；舱内永远有高亮的"建议动作"（规则引擎给的默认选项）。
- **落点**：新容器 `src/components/layout/IncidentRoom.tsx`（通用外壳，后续结算差异/合约违约复用）+ `src/pages/workspaces/RiskIncident.tsx`；影响预演纯前端由 store 号池权重计算；复盘卡写 activity。
- **验收**：从总览待办一键开舱、处置、回原位（滚动位置保持）；预演数字与号池权重口径一致（单测）。

### 2.3 行动队列（待办的制度化升级）🔴

- **原因**：现在的"今日待办"点完就消失，没有认领/延后/追溯——多人协作时两个人处理同一单都不知道。
- **想法**：待办升级为四态（待处理/已认领/已延后/已完成），认领人+时间可见，完成自动写审计与事件流；数据结构进 store（演示）/新表 ActionItem（真实，B2 再接）。
- **小白形态**：简洁总览的"今天三件事"（1.2）就是行动队列的头部三条——**同一数据，两种呈现**。
- **落点**：`store.ts` 加 actions 集合与流转函数；Dashboard/处置舱/结算工作台统一从它取数。
- **验收**：认领互斥（先到先得）；延后项次日回队；完成项可在审计检索。

### 2.4 ⌘K 动作化 🟠

- **原因**：熟手的效率天花板在点击深度。退款一单现在要 订单页→找到行→抽屉→按钮→确认 5 步。
- **想法**：CommandPalette 加动作注册表：`退款 O-99812`、`熔断 M-BL-01`、`开始结算`、`切暗色`、`去成员页邀请`。动作 = {匹配语法, 权限点, 执行函数(复用现有 store/api), 是否需确认}。**有权限才出现**，资金类动作弹标准 Confirm（不绕过任何守卫）。
- **小白形态**：对小白不可见不打扰（不用就永远不知道）；面板底部一行灰字"试试输入：退款 订单号"做渐进式引导。
- **落点**：`src/components/layout/CommandPalette.tsx` 加 actions 数组 + 参数解析；首批 10 个动作。
- **验收**：10 动作全部走权限校验与确认；无权限用户搜不到对应动作。

### 2.5 订单页三栏工作台 + 补齐筛选 🔴

- **原因**：评审明确指出：一个订单页居然没有日期筛选、没有搜索、没有分页 UI——后端游标分页早就支持，前端一直没接。这是最基本的欠账。
- **想法**：列表(可筛可搜) | 详情(状态机时间线+关联工单/订阅) | 动作面板 三栏（<lg 折叠为列表→抽屉）；顶部日期范围 + 类型 + 关键词；"我的视图"可保存筛选组合（prefs 持久化）。
- **小白形态**：简洁模式三栏收为两栏（列表+详情），动作只留"退款"一个主按钮；5 列表格（1.3 契约）。
- **落点**：`src/pages/Orders.tsx` 重构；筛选参数接 `bizApi.orders` 已有的 query（type/dateFrom/dateTo/cursor）；详情栏复用 OrderDrawer 内容组件化。
- **验收**：筛选/搜索/加载更多全走服务端参数（真实模式）；视图保存/恢复；键盘 ↑↓ 切换选中行。

### 2.6 分润 Sankey 资金流向图 🟠

- **原因**：`gross = 品牌留存 + 平台费 + 代理分润 + 准备金 + 冲账` 这条恒等式是整个平台的灵魂，但它只活在代码和对账报告里——**最该可视化的东西没有可视化**。
- **想法**：结算页新增"资金流向"视图：一张自研 SVG Sankey（左 GMV 一根粗流，右分五股，宽度=金额，对账差异在对应股上标红脉冲）。悬停显示金额与占比，点击股 → 该项明细。
- **小白形态**：这张图**就是**给小白的——一眼看懂钱怎么分的，比五列数字直观一个量级。简洁模式默认展示图，专家模式默认表格+图切换。
- **落点**：`charts.tsx` 加 `Sankey` 组件（单层分流，无需通用 DAG，约 150 行）；`Settlement.tsx` 加视图切换。
- **验收**：五股宽度之和恒等于 gross（渲染前断言）；明暗主题下对比度达标；reduced-motion 无脉冲。

## 三、B2 · 实时化与页面重造（2-3 周量级）

### 3.1 SSE 事件通道 🔴

- **原因**：页面上写着"实时"，实际是 5 分钟轮询——演示时观众看不到"活着"的感觉，生产时运营错过黄金处置窗口。
- **想法**：`GET /events/stream`（SSE，够用且比 WebSocket 少一整套连接治理）：订单/风控/通知三类事件。**协议一套、来源两个**——真实模式服务端推，演示模式本地定时器产同格式事件（`src/lib/liveDemo.ts`），前端 `store.applyEvent()` 增量合并，永不整页刷。
- **小白形态**：无感知成本——只是数字自己会动了、事件流自己会长了。新事件行用 diff-flash 微动效提示（reduced-motion 跳过）。
- **落点**：后端 `server/src/common/events.controller.ts`（Nest SSE + 权限 scope 过滤：品牌只收自己的事件）；前端 `store.ts` 加 applyEvent + 断线指数退避重连。
- **验收**：端到端延迟 <2s；断网 30s 重连后无重复无丢失（事件带自增 id，Last-Event-ID 续传）；品牌门户收不到他牌事件（e2e）。

### 3.2 风控中心重设计：静态贴图 → 规则引擎工作台 🔴

- **原因**：评审最重的批评——SIGNALS/EVENTS 是冻结常数（写着 64 条规则，库里只有 6 条），页面好看但是假的。**假实时比没有实时更伤信任。**
- **想法**：左=规则列表（真实 CRUD + 启停 + 每条规则的真实命中计数），右=实时事件流（接 3.1），顶部拦截漏斗由事件真值聚合。规则命中计数在演示模式由 liveDemo 事件驱动累加——数字全部可溯源。
- **小白形态**：简洁模式只显示"三道防线状态卡"（每道防线：正常/告警 + 一句话）+ 事件流；规则编辑收进专家模式。
- **落点**：`src/pages/Risk.tsx` 重写数据源（删除全部冻结常数）；规则 CRUD 已有 store 函数，补服务端 RiskRule 表与端点（真实模式）。
- **验收**：页面零静态业务数字；每个数字点击可下钻到来源。

### 3.3 资金合规页：幻灯片 → 检查清单 🟠

- **原因**：现在是政策长文卡片，读一次就再也不会打开——合规应该是**持续运营的状态**，不是一篇文章。
- **想法**：每条红线一行：`状态(达标/风险/未办) · 证据（链接到审计/文档）· 责任人 · 复查日期`，可勾检可留言，状态持久化；政策原文折叠为引用。二清红线、三流一致等 8 项先行。
- **小白形态**：清单形态天然小白友好——红黄绿状态灯 + 到期提醒进行动队列。
- **落点**：`src/pages/Compliance.tsx` 重构；数据结构进 store/Config。
- **验收**：8 项红线全部有状态与复查日期；到期自动生成待办。

### 3.4 对账解释器（确定性算法，不是 AI）🔴

- **原因**：差异单只给一个数字"¥31,200"，解释靠人翻流水——这是运营最耗时的单点。**它看起来像 AI 魔法，实际是集合求差，先做最实用的。**
- **想法**：服务端把差异分解为构成项：`差异 = Σ跨期退款 + Σ通道费口径差 + Σ未回传单`，每项附明细单号列表与建议动作（核销/催回传/人工复核）。
- **落点**：`server/src/business/reconciliation.service.ts` 加 `explainDiff(settlementId)`；结算工作台②步与清结算页差异行内嵌展示。
- **验收**：种子数据的 2 笔差异 100% 给出构成；构成项金额之和 === 差异额（断言）；e2e 覆盖。

### 3.5 接入进度轨道 + 代理成长面板 🟠

（v9 骨架 1.3/1.4 照做，补充执行细节）

- **落点**：进度轨道 = `src/pages/BrandDetail.tsx` 顶部组件 + `portal/BrandPortal.tsx` 首页卡，状态由真实数据派生（有无商户号/在架商品/首单/RSA 就绪），**不新增状态字段**——派生的进度永远不会和现实打架，这是想法的关键。
- 成长面板 = `portal/AgentPortal.tsx` 信用分页重构：四维雷达（自研 SVG，`charts.tsx` 加 Radar ~80 行）+ 升级条件进度条 + 层级权益表（选品市场按层过滤真实生效）。
- **小白形态**：两者本质都是"下一步指引"的具象化，天然 L0。

### 3.6 异动播报（规则模板起步）🟠

- **原因**：数据会说话但没人翻译。晨报是小白理解大盘的最短路径。
- **想法**：规则引擎起步（环比阈值 + 文案模板："{品牌}退款率环比+{pct}%，集中在{商户号}，建议{动作}"），生成 3 句话晨报置顶事件流；`INSIGHT_PROVIDER=rule|llm` 留插槽，**LLM 只改写文案不改数字**（数字永远来自规则引擎，防幻觉）。
- **落点**：`src/lib/insights.ts`（纯函数：store → 洞察数组）；总览与事件流消费。
- **验收**：晨报每句都带下钻链接；数字与页面口径一致（单测比对）。
