# 下一轮升级规划 v9 — 从「正确的系统」到「顺手的产品」

> v8 解决的是**正确性**：资金不双花、权限不越界、演示不掉链、明暗两套皮肤。
> v9 要解决的是**顺手**：运营完成一件事要跨几页、点几次、想几秒 —— 把「页面的集合」升级为「动线的产品」。
> 三个关键词：**动线化**（任务导向重构交互）· **实时化**（假实时变真实时）· **智能化**（AI 从素材玩具变运营副驾）。

---

## 〇、v8 之后我们站在哪（底盘盘点）

| 维度 | 现状 | 对 v9 的意义 |
|---|---|---|
| 正确性 | 五轮对抗审查、60+ 缺陷修复、190 测试、资金恒等式对账 | 可以放心在上面做体验层创新，不会盖在沙子上 |
| 设计系统 | 令牌单一来源 + 明/暗/跟随系统三态，全站零硬编码色 | 白标（品牌换肤）、密度切换的技术地基**已经打好** |
| 三类入口 | 控制台 19 页 + 双门户 19 页 + C 端超市，演示模式全通 | 动线重构不需要新建页面，需要**重新连接**页面 |
| 数据层 | store 水合镜像 + 106 API + 幂等/审计/RBAC 三级 | 实时化只差一个推送通道，写路径全部就绪 |
| 弱点（评审已指出） | Risk 页静态感重、Orders 无筛选分页、Compliance 像幻灯片、跨页任务靠人脑记进度、"实时"是定时刷 | **这就是 v9 的靶子** |

## 一、动线设计（v9 的主轴）

> 设计原则：**运营的一天不是按页面组织的，是按任务组织的。**
> 每条动线给出：现状痛点 → 目标动线 → 具体落地改动 → 度量口径。
> 全部动线共享三个新机制（见 1.6）：行动队列 · 处置舱 · 可执行命令面板。

### 1.1 结算日动线（最高频资金任务）🔴 P0

**现状痛点**：月结日运营要在 清结算 → 对账中心 → 准备金台账 → 代理提现 四处横跳，哪些单子核过、哪些差异没解释，全靠脑子记；中途被投诉打断回来就断片。

```
现状：清结算页 ──(人肉记忆)──> 对账差异 ──(切页)──> 核销 ──(切页)──> 准备金释放 ──(切页)──> 提现审批
目标：
┌─ 结算工作台（Checklist 模式，一屏走完）───────────────────────┐
│ ① 拉取对账 ▸ 已平 6 / 差异 2        [逐笔展开差异归因]      │
│ ② 处理差异 ▸ T-5516 核销 ✓ · S-2405-MG 挂起(留言)          │
│ ③ 发起结算 ▸ 待结算 2 笔 ¥168.3万   [批量确认·二次确认]     │
│ ④ 释放到期准备金 ▸ 3 笔 ¥53.9万     [一键释放·守恒校验]     │
│ ⑤ 审批提现 ▸ 代理 2 单               [逐单批/驳]            │
│ ──────────────────────── 本期结算完成 · 生成结算报告 ────────│
└──────────────────────────────────────────────────────────────┘
```

**落地改动**
- 新增 `/settlement/run` 结算工作台：五步 Checklist，每步就是现有 API 的编排（零新后端），步骤状态持久化（中断可续）
- 差异行内嵌「归因解释器」：自动比对退款流水 ↔ 冲账记录，给出差异构成（见 3.4 智能对账）
- 完成后生成**本期结算报告**（结构化摘要 + CSV 附件），写入审计并可推送品牌门户通知

**度量**：结算日动线完成时长（目标 < 10 分钟）；中断续跑率；差异未解释残留数

### 1.2 风险处置动线（告警 → 复盘闭环）🔴 P0

**现状痛点**：总览待办点「处理」跳到风控页后上下文丢失；处置完成后没有「因为这次处置，投诉率降了吗」的回看。

```
告警卡片 ──> 【处置舱 Focus Mode】────────────────────────────┐
             │ 左：事件上下文（商户号曲线·关联订单·同代理近7天） │
             │ 中：处置动作（熔断/切量/降权/退款/驳回，带影响预演）│
             │ 右：处置留痕时间线（谁·何时·做了什么·结果）      │
             └──> 完成后 48h 自动生成复盘卡（指标前后对比）──────┘
```

**落地改动**
- 「处置舱」= 全屏 Modal 路由（`/risk/incident/:id`）：聚合该事件的商户号健康曲线、关联订单流、代理画像于一屏，动作按钮带**影响预演**（如"熔断 M-BL-01 → 预计 ¥42万/日 流水改道 M-MG-02"）
- 待办卡片直接开舱，处置完回原位（滚动位置保持）
- 复盘卡：处置后 48h（演示态：立即）对比处置前后投诉率/流水，落入实时联动事件流

**度量**：告警 → 处置完成时长；处置复发率（同商户号 7 天内二次告警）

### 1.3 品牌接入动线（增长侧第一印象）🟠 P1

**现状痛点**：新品牌从入驻到首单要跨 品牌页 → 配置 → 号池 → 商品 → 合约 五处，没有进度概念；门户侧品牌方不知道"下一步该干嘛"。

```
[平台侧] 入驻审核 ─┬─> 接入进度轨道（Progress Rail，常驻品牌详情页顶部）
                   │   资质✓ → 费率/账期✓ → 商户号(1/2) → 商品上架(0) → 首单验证(未)
[品牌门户侧] 首页新手任务卡 ──> 「完成接入还差 2 步」──> 直达对应门户页
```

**落地改动**
- 品牌详情页顶部加**接入进度轨道**（五段式，由真实数据派生：有无商户号/在架商品/首单）
- 品牌门户首页加**下一步卡片**（同一派生逻辑的门户视角），完成全部步骤后卡片变为经营周报入口
- 开发者中心「就绪度体检」并入轨道第五段（RSA 密钥/回调地址就绪才算接入完成）

**度量**：品牌入驻 → 首单时长（Time-to-First-Order）；接入完成率

### 1.4 代理成长动线（门户留存）🟠 P1

**现状痛点**：代理门户是"查数据的地方"，不是"变强的地方"。信用分只展示不解释，升级 L2 的路径不可见。

```
L1 新代理 ──> 成长面板：信用分构成雷达（履约/投诉/退款/活跃 四维）
          ──> 「距 L2 还差：信用分 +38 · 连续 30 天投诉率 <0.5%」
          ──> 达标自动升级 → 解锁更高分润档 + 更多可投品牌 → 事件流恭喜 + 通知
```

**落地改动**
- 代理门户「信用分」页升级为**成长面板**：四维构成、升级条件进度条、历史信用分曲线（现有数据即可派生）
- 层级权益表（L1/L2/L3 分润与准入差异）从文案变成真实生效的配置（选品市场按层过滤）

**度量**：代理周活跃（登录+领链接）；L1→L2 转化率

### 1.5 C 端购买动线（补最后一公里）🟡 P2

**现状痛点**：超市支付成功后动线断了——"凭套餐号查看进度"无处可查。

```
组合 → 支付 → 【我的订阅 /market/me】：套餐号查询（免登录，号+手机尾号）
              ├─ 开通进度：拆单 3 笔 · 有道✓ 芒果⏳ 喜马拉雅⏳
              ├─ 每笔订阅状态（激活/续费日/金额）
              └─ 退订入口（合规要求：路径不得超过 3 步）
```

**落地改动**
- 新增 `/market/me`：按 bundleId 查询开通进度（复用 fulfillment 数据；演示态本地合成）
- 支付成功页直接给进度页链接 + 二维码占位
- 合规退订三步路径（选择原因 → 挽留报价 → 确认），打通已有的退订/挽留/win-back 状态机展示

**度量**：支付 → 首次回访进度页比例；退订路径完成率（合规指标，越顺畅越好）

### 1.6 全局动线机制（三件横切基建）🔴 P0

| 机制 | 是什么 | 替代什么 |
|---|---|---|
| **行动队列 Action Queue** | 待办升级：每项待办有 认领/延后/完成/留言 四态，跨会话持久，完成自动写审计与事件流 | 现在的"今日待办"只能点一下就消失，无法协作与追溯 |
| **处置舱 Incident Room** | 全屏聚焦模式的通用容器（1.2 的风控款是第一个租户），后续结算差异、合约违约复用同一外壳 | 现在的 DetailPopover 承载不了多信息源处置 |
| **可执行命令面板** | ⌘K 从"搜索"升级为"动作"：`退款 O-99812`、`熔断 M-BL-01`、`切暗色`、`去结算工作台` —— 动作复用现有 API + 权限校验 + 二次确认 | 高频操作从 3-4 次点击降到一次键入 |

## 二、页面设计升级（Design v3）

> 原则不变：精密仪表风格、令牌单一来源、克制动效。v3 解决的是**信息架构层**的问题，不是换皮。

### 2.1 版式范式：三栏工作台（高频页收敛）

评审指出的 popover 过载问题（详情、动作全塞气泡）在高频页改为**列表-详情-动作三栏**：

```
┌──────────┬────────────────────────┬──────────────┐
│ 列表(可筛) │ 详情（选中行的全上下文）   │ 动作面板       │
│ 订单流    │ 订单状态机时间线          │ 退款并冲账     │
│ ▸O-99812 │ 关联工单/合约/订阅        │ 加入行动队列   │
│  O-99811 │ 同代理近 7 天异常         │ 复制对账链接   │
└──────────┴────────────────────────┴──────────────┘
```

- 首批改造：**订单页**（+日期范围/关键词搜索/游标分页 UI——后端早就支持，前端一直没接）、**工单页**、**代理页**
- 低频页（审计、配置）保持现有单栏；DetailPopover 保留给轻量速览
- 响应式：<lg 时三栏折叠为 列表→抽屉 两级

### 2.2 页面级重设计清单（对着评审弱页 Top5 开刀）

| 页面 | 现状问题 | v3 设计 | 优先级 |
|---|---|---|---|
| **风控中心** | 静态贴图感：SIGNALS/EVENTS 是冻结常数，与 store 里 6 条规则矛盾 | 改为**规则引擎工作台**：左规则列表（真实 CRUD + 启停 + 命中计数），右实时事件流（接 3.1 推送）；顶部拦截漏斗由事件真值聚合 | 🔴 |
| **订单页** | 无日期筛选/无搜索/无分页 UI，KPI 与表格曾脱节 | 三栏工作台首发页（见 2.1）+ 保存筛选视图（"我的视图"chips） | 🔴 |
| **资金合规** | 像幻灯片：政策长文 + 常数 KPI | 改为**合规检查清单**：每项红线一行（状态·证据链接·责任人·复查日期），可勾检状态持久化；政策原文折叠为引用 | 🟠 |
| **增长合约/置换/商品** | 三页同构但风格漂移，demo 态曾显示为一片 0 | 统一「交易台」模板：状态漏斗头 + 三栏列表 + 统一的发起 Wizard 骨架 | 🟠 |
| **数据·归因** | 图表只能看不能问 | 交互 2.0（见 2.3）+ 维度下钻（点漏斗层 → 该层订单列表） | 🟡 |

### 2.3 数据可视化 2.0（继续零依赖自研 SVG）

- **交互三件套**：区间刷选（brush，选一段时间联动全页 KPI）· 对比模式（本期 vs 上期虚线叠加）· 点击下钻（图 → 明细列表）
- **分润 Sankey 图**：结算页新增「资金流向」视图 —— GMV → 品牌留存/平台费/代理分润/准备金/冲账 的可视化恒等式（对账差异直接在图上标红），这是这个产品最该有而没有的一张图
- **号池热力图**：商户号 × 日期 的投诉率热力矩阵，熔断/切量决策一眼看出模式
- **留存曲线预测带**：cohort 曲线尾部加预测区间（配合 3.3 LTV 预测）

### 2.4 密度 / 空态 / 移动端 / 可访问性

| 项 | 内容 |
|---|---|
| 密度系统 | 顶栏加 舒适/紧凑 切换（表格行高 44→36px，令牌 `--row-h` 一处控制）——财务对账场景强需求 |
| 空态系统 | 统一 EmptyState 语言：图标 + 一句为什么空 + 一个下一步动作按钮（现在部分页有部分页没有） |
| 移动端 | 门户优先响应式（品牌/代理老板在手机看数据、批提现）；控制台保证只读浏览可用，处置舱桌面独占 |
| 可访问性 | 键盘走完五条核心动线（Tab 顺序审计）；对比度批量校验脚本进 CI（明暗两套都跑）；表格加 `scope`/`caption` |
| 微动效 2.0 | 数字变更 diff-flash（旧值划出新值滑入）；行插入 FLIP；页面切换接 View Transitions API（渐进增强，reduced-motion 全跳过） |

## 三、智能化（AI 从玩具到副驾）

> 边界先说死：**AI 只读数据 + 起草建议，资金动作永远人来点**。所有 AI 输出带"依据"链接，可溯源到订单/工单原始数据。

| # | 能力 | 形态 | 落地方式 | 优先级 |
|---|---|---|---|---|
| 3.1 | **异动播报** | 总览"实时联动"升级：不止记录事件，还生成 3 句话晨报（"芒果 TV 退款率环比 +40%，集中在 M-MG-02 银行分账通道，建议核查"） | 规则引擎起步（阈值+环比模板），LLM 接口留插槽（`INSIGHT_PROVIDER=rule\|llm`） | 🔴 |
| 3.2 | **Ask 平台** | ⌘K 里直接问："上周喜马拉雅退了多少钱""哪个代理投诉率涨最快" → 结构化查询 + 图表卡回答 | 意图 → 预置查询模板映射（先 20 个高频问法，不做开放 NL2SQL——防幻觉也防越权），权限沿用提问者 scope | 🟠 |
| 3.3 | **LTV 预测** | cohort 曲线拟合预测 D90 净 LTV，指导合约定价（"该品牌 LTV 走弱，建议 reservePct 上调") | 纯前端可算（指数衰减拟合），预测带画进留存图 | 🟠 |
| 3.4 | **对账解释器** | 差异单不再只给数字：自动匹配"差 ¥18,400 = 3 笔跨期退款 + 1 笔通道费口径差"并给出核销建议 | 服务端确定性算法（流水集合求差），不是 AI 但长得像魔法——先做这个，最实用 | 🔴 |
| 3.5 | **投诉根因聚类** | 工单按 reason 文本聚类出 Top 主题（"自动续费未告知"类 ×12↑），直接挂到风控规则建议 | 关键词规则起步，向量聚类留插槽 | 🟡 |
| 3.6 | **AIGC 素材闭环** | 现有素材实验台补上"生成 → 投放 → LTV 回流 → 自动淘汰/加码建议"的最后一环 | 素材 LTV 排名已有，加建议引擎（规则）+ 一键复制优胜素材参数 | 🟡 |

## 四、实时化与协同

| # | 内容 | 说明 | 优先级 |
|---|---|---|---|
| 4.1 | **SSE 事件通道** | `GET /events/stream`（Server-Sent Events，比 WebSocket 少一层运维复杂度够用了）：订单流/风控事件/通知实时推送。前端 store 增加 `applyEvent()` 增量合并，替代"每 5 分钟刷新"的假实时。演示态用本地定时器模拟同一事件协议——**协议一套，来源两个** | 🔴 |
| 4.2 | **在线痕迹 presence** | 工单/结算单显示"陈风控正在处理"（SSE 心跳 + 60s 过期），行动队列认领冲突即时可见 | 🟠 |
| 4.3 | **乐观锁提示** | 关键实体加 version 字段，镜像写冲突时不再静默回收，而是给 diff 弹窗（"该工单已被王运营改为已解决，是否仍要升级？"） | 🟠 |
| 4.4 | **对外 Webhook 总线** | 品牌可订阅平台事件（结算出单/投诉升级/商品过审），复用有道对接已有的 RSA 签名投递器与重试日志——把单点集成泛化为平台能力 | 🟡 |

## 六、批次排期（三批，每批可独立交付）

## 七、度量体系（v9 自己的北极星）

| 层 | 指标 | 现状 | 目标 |
|---|---|---|---|
| 动线 | 结算日动线完成时长 | 无法度量（跨页无埋点） | <10 分钟，全程可中断续跑 |
| 动线 | 告警→处置完成时长 / 处置复发率 | 同上 | 处置 <3 分钟；复发 <10% |
| 体验 | 高频操作点击深度（退款/熔断/结算） | 3-5 次点击跨页 | ⌘K 一步或工作台内 ≤2 次 |
| 实时 | 事件端到端延迟 | ~5 分钟（轮询） | <2 秒（SSE） |
| 性能 | 首包体积 / LCP | 单包全量 | <180KB gzip / LCP <1.5s |
| 质量 | 视觉回归覆盖 | 0（手工看） | 12 页 × 明暗 2 主题进 CI |

埋点方案：轻量自建（`track(event, props)` → 演示态入 store 可视化，真实态入审计旁路表），**不接三方统计**（数据不出域，合规姿态一致）。

## 八、不做清单（与红线）

- ❌ 刷单/虚假交易/绕支付风控——继续绝不实现（《电商法》17 条红线，合规页已列）
- ❌ 开放式 NL2SQL——Ask 平台只走预置模板，防幻觉、防越权、防注入
- ❌ 重型图表库/组件库引入——自研 SVG 体系继续，保持包体与风格控制力
- ❌ WebSocket 全双工——SSE 单向推送够用，少一套连接管理与鉴权面
- ❌ AI 直接执行资金动作——AI 只读 + 建议，资金键永远在人手上
- ❌ 微前端/服务拆分——单体仍是当前规模的最优解，过早拆分只添运维债
