## 0. 现状与现实约束（必须先讲清）

- **现状**：本项目是**纯前端原型**（Vite + React + TS），数据在 `lib/store.ts` 的 localStorage mock store。**没有后端服务、没有数据库、没有真实鉴权**。
- **L3 的本质**：真实登录态 / Token / 服务端 RBAC / 审计落库，需要一个**真实后端服务 + 数据库 + 部署环境**。这是与当前前端原型**并列的新工程**，无法用纯前端实现，也无法在当前沙箱里跑通生产后端。
- **因此本轮交付**：① 这份可落地的完整方案（架构/鉴权/RBAC/审计/后台清单/API/数据模型/安全/对接/路线图）；② 可执行的下一步三选（见 §13）。
- **不做**：在没有真实后端的前提下假装"已上线"。资金/支付/分账的真实对接以持牌机构与法税意见为准。

## 2. 技术架构与选型（建议）

**前后端分离**，前端复用现有 React 应用（改数据层指向 API）。

| 层 | 建议选型 | 理由 |
|---|---|---|
| 后端框架 | **Node.js + NestJS**（或 Express + ts） | 与前端同语言(TS)，团队上手快；NestJS 自带模块化/守卫/拦截器，天然适配 RBAC + 审计 |
| ORM / DB | **Prisma + PostgreSQL** | 类型安全、迁移规范；Postgres 适配多租户、JSON 字段、审计 |
| 鉴权 | **JWT(access) + 刷新令牌(refresh, httpOnly cookie)** | 无状态、可水平扩展；refresh 放 httpOnly 防 XSS 窃取 |
| 缓存/队列 | Redis（会话黑名单、限流、异步审计/通知队列） | 登出失效、限流、削峰 |
| 部署 | 容器化(Docker) + 反向代理(Nginx) + HTTPS | 标准化、可灰度 |
| 可选本地起步 | **SQLite + Prisma** | 零依赖本地可跑的 starter，便于先验证（见 §13） |

> 也可选 Go(Gin)/Java(Spring) 等；建议 TS 全栈以降低协作成本。

## 3. 账户 / 鉴权 / 个人账户

| 能力 | 说明 |
|---|---|
| 登录 | 账号(手机号/邮箱) + 密码；密码 **bcrypt/argon2 哈希**，绝不明文 |
| 会话 | 登录返回短时 **access JWT**（~15min）+ **refresh token**（httpOnly cookie，7–30天）；access 过期用 refresh 静默续期 |
| 登出 | 撤销 refresh（Redis 黑名单），前端清除 access |
| 双因子(2FA) | 可选 TOTP / 短信验证码（管理员强制） |
| 找回/改密 | 邮箱/短信验证码重置；改密后失效所有会话 |
| 个人资料 | 头像、姓名、联系方式、所属角色（只读） |
| 登录安全 | 登录设备/IP 列表、异地登录提醒、失败锁定（防爆破） |
| 防护 | 登录限流、验证码、CSRF（cookie 模式下） |

**前端**：未登录访问任意页 → 重定向到 `/login`；登录态存内存 + refresh cookie；路由守卫 + 401 自动跳登录。

## 4. RBAC 权限模型（角色 / 权限点 / 数据范围）

三级控制：**路由级**（能不能进页面）+ **操作级**（能不能点按钮/调接口）+ **数据级**（能看到哪些数据）。

**预设角色（与现有"配置中心"一致并扩展）**

| 角色 | 范围 |
|---|---|
| 平台超级管理员 | 全部权限 |
| 财务 / 清结算 | 结算、对账、提现、发票、资金台账 |
| 风控 / 售后 | 风控规则、工单、退款、名单、号池干预 |
| 运营 | 品牌、代理、选品、素材、数据 |
| 品牌方(租户) | **仅自己品牌**的数据与配置（数据级隔离） |
| 代理商(租户) | **仅自己**的链接、消耗、转化、分润、提现 |
| 只读审计 | 全部只读 + 导出 |

**权限点（permission，示例）**：`brand.read / brand.review / settlement.clear / withdrawal.approve / merchant.fuse / ticket.refund / member.invite / config.write / audit.read …`

**权限矩阵**：角色 × 权限点的可勾选表（后台可视化编辑）。
**数据范围**：每用户绑定 `scope`（platform / brand:{id} / agent:{id}），后端在每个查询注入过滤（品牌只见自己、代理只见自己——已是 v1/v2 既定原则）。
**前端**：菜单与按钮按权限点 **可见/禁用**（无权限的入口不渲染或置灰）。

## 5. 操作审计 / 系统日志

- **审计事件模型**：`{id, actor, role, action, resource, resourceId, before, after, ip, ua, at}`。
- **必留痕范围**：所有**资金类**（结算/冲账/提现/退款）、**权限类**（角色/成员/权限矩阵变更）、**配置类**（阈值/通道/费率/路由）、**风控处置**（拉黑/冻结/熔断）。
- **实现**：NestJS 拦截器统一捕获写操作 → 异步写审计表（不阻塞主流程）。
- **后台页**：按 操作人/动作/资源/时间 检索；diff 展示 before→after；导出 CSV。
- **防篡改**：审计表只追加（append-only）、可加哈希链。
- 前端已有的 `activity` 事件流可作为审计的展示雏形，后端化后落库。

## 8. API 契约（按资源）

**通用约定**
- 鉴权头：`Authorization: Bearer <access>`；refresh 走 httpOnly cookie。
- 列表：`?page&pageSize&sort&filter`（cursor 分页可选）；统一响应 `{ data, total }`。
- 写操作：带 `Idempotency-Key`；资金类操作要求二次确认 token。
- 错误码：`401 未认证 / 403 无权限 / 409 冲突(幂等) / 422 校验失败`，统一 `{ code, message }`。

**鉴权**
```
POST /auth/login            {account, password} → {access, user, scope, permissions}
POST /auth/refresh          (cookie) → {access}
POST /auth/logout
GET  /auth/me               → {user, role, scope, permissions}
POST /auth/password/reset   {token, newPassword}
```

**RBAC**
```
GET    /members           列表    POST /members 邀请
PATCH  /members/:id       停用/改角色
GET    /roles             POST /roles    PATCH /roles/:id (权限点)
GET    /permissions       权限点字典
```

**审计**
```
GET /audit-logs?actor&action&resource&from&to   → 分页
GET /audit-logs/export
```

**业务资源（每个含标准 CRUD + 状态流转端点，承袭 v2 §5.1）**
```
brands / plans / channels / merchantAccounts / agents / orders / subscriptions
trackLinks / settlements / ledger / withdrawals / invoices / tickets
riskRules / riskEvents / lists / reconciliations / attributionConfig / configs / webhooks
```
示例：`POST /settlements/:id/clear`、`POST /tickets/:id/refund`、`POST /merchants/:id/state`、`POST /withdrawals/:id/approve`、`POST /brands/:id/review`。

## 10. 安全与合规

- 密码 **argon2id 哈希** + 盐；绝不存明文/可逆。
- access JWT 短时；refresh httpOnly + Secure + SameSite；登出/改密失效会话。
- 传输全程 **HTTPS**；CSRF 防护（cookie 模式）；XSS 输出转义。
- **加密脱敏**：手机号/支付信息存储加密、展示脱敏（遵《个人信息保护法》）。
- **最小授权 + 数据级隔离**：品牌只见自己、代理只见自己。
- 资金/不可逆操作：**二次确认 + 强制审计 + 幂等**。
- 限流（登录、敏感写）、监控告警、灾备、合规取证留痕。
- 资金合规：过路只走持牌分账、不自建资金池（承袭 v1 §3，红线不变）。

## 11. 前端如何对接（改造点，保留现有 UI 投入）

1. **抽象数据层**：把 `lib/store.ts` 的直接读写改为 `lib/api.ts` 客户端；用环境变量切换 **mock 模式 / 真实 API 模式**（现有演示态保留，接后端时切换）。
2. **登录态**：新增 `/login` 页 + AuthContext；access 存内存、refresh cookie；axios/fetch 拦截器自动带 token + 401 跳登录 + 静默刷新。
3. **路由守卫**：未登录拦截；按 **权限点** 渲染侧边栏菜单与页面内操作按钮（无权限不渲染/置灰）。
4. **数据范围**：登录后拿到 scope/permissions，前端据此过滤可见入口；真实过滤以后端为准（前端只是体验层）。
5. 现有 13 页改为调 API；状态机/联动由后端权威驱动（前端展示）。
