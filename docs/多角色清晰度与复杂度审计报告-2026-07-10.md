## 审计结论

本轮覆盖 8 个正式角色和 1 个故意制造的角色/范围错配账号。基线与收口审查发现 3 个 P1、5 个 P2 问题，均已修复并完成同路径回归。

修复后的核心变化不是继续增加页面，而是让同一套系统按职责收敛：平台超管保留全局经营视角；财务、风控、运营、审计只看到本职指标和可达动作；团队管理员只做成员状态与审计；品牌方和代理商先看到日常任务，低频工具按需展开。

综合健康度：**可进入主仓**。修复后未发现残留越权或资金回归，浏览器回归无 error/warning，390px、768px 和桌面宽度均无页面级横向溢出。

| 角色 | 首屏任务 | 本轮验证结果 |
|---|---|---|

| 财务 / 清结算 `finance` | 对账、结算、提现与准备金 | 首屏改为财务工作台；只显示财务指标和清结算入口 |

| 团队管理员 `teamadmin` | 停用/恢复成员、查看操作审计 | 移除邀请和角色编辑；状态更新真实成功，超级管理员显示为受保护账号 |

| 错配夹具 `brandaudit` | 角色为审计、范围为品牌 | 不再进入空导航和错误页；明确提示角色与范围不匹配并可退出 |

## 试用步骤

每个正式角色都执行了以下主路径；涉及动作的角色继续完成一次落地验证。

1. 从对应入口登录，确认身份、角色、环境标识和首页标题一致。
2. 扫描首屏，确认用户能在不理解全平台模型的前提下回答“我今天要做什么”。
3. 检查侧栏、搜索提示、健康信号和待办是否只暴露有权访问的目标。
4. 点击主按钮和异常动作，确认进入正确工作台，不出现跳回首页的权限死路。
5. 用团队管理员停用并恢复审计账号，确认只提交状态差异、不夹带角色变更。
6. 展开团队角色矩阵，确认保存按钮和 28 个权限点对团队管理员均为只读。
7. 在品牌方和代理商门户检查默认导航数量、折叠组展开行为和首屏主动作。
8. 用 390px 移动端检查品牌、代理、团队管理员，用 768px 检查风控工作台。
9. 检查页面级横向溢出、浏览器 error/warning 和角色错配的失败状态。

## 发现与修复

| 严重度 | 基线问题 | 用户影响 | 修复与回归 |
|---|---|---|---|
| P1 | 内部首页对财务、风控、运营、审计复用超管叙事和动作 | 用户看到无关指标；无权链接点击后跳回首页；无权限数据以 0 显示，容易被误判为健康 | 新增角色体验配置，按权限过滤或重定向目标；各角色使用本职标题、说明、指标和主动作 |
| P1 | 团队管理员前端显示邀请、改角色和保存权限，但后端只允许超管 | 界面承诺与真实权限冲突；状态更新因夹带 `roleId` 返回 403 | 隐藏邀请；角色矩阵只读；成员弹窗只发送真实变化字段；停用和恢复浏览器实测成功 |
| P1 | 团队管理员可绕过界面直接停用超级管理员 | 最高权限账号可能被低层级管理员锁定 | 服务端把 `super` 目标设为不可经 API 修改；成员列表显示“受保护账号”；新增 E2E 断言 403 |
| P2 | 品牌门户默认展开 13 个入口、5 组能力混在一起 | 移动端滚动长，日常任务被增长、合作、接入设置淹没 | 日常工作默认展开，增长/合作/接入折叠；默认可见入口降为 5 个 |
| P2 | 品牌方、代理商首页只有结果，没有一个明确起点 | 新用户能看数据但不知道下一步 | 品牌新增“管理商品”，代理新增“去选品投放”，均验证到达目标页 |
| P2 | 本地真实 API 模式仍显示“生产环境” | 环境判断错误，可能造成误操作心理预期 | 徽标按真实状态显示“生产环境 / 真实接口 / 演示数据” |
| P2 | `brandaudit` 错配账号进入空侧栏和错误边界 | 用户无法判断是权限、数据还是系统故障 | 无可用门户权限时显示明确原因、联系对象和退出动作 |
| P2 | 通用 `PageHeader` 在 390px 下仍把说明和主按钮挤在同一行 | 说明变窄，按钮干扰阅读顺序 | 共享页头移动端纵向排列；团队管理员实测说明底部与按钮顶部间距 12px、横向溢出 0 |

## 复杂度控制

本轮采用“按角色减法”，没有新增页面、路由或依赖：

- 角色差异集中在 `roleExperience.ts`，首页不再散落不可达的角色分支。
- 导航继续以服务端权限为边界，只在前端做任务排序和默认展开状态。
- 团队成员更新抽成纯函数，仅提交变化字段，避免 UI 与后端契约再次漂移。
- 门户保留全部能力，但低频组默认折叠，不用删除功能来换取简洁。
- 移动端问题修在共享 `PageHeader`，没有为单个页面增加特例。

## 可访问性与限制

折叠导航使用语义化按钮和 `aria-expanded`；移动菜单提供明确的打开/关闭名称；权限矩阵的只读状态由真实 `disabled` 属性表达。本轮覆盖键盘可识别语义、窄屏布局和主要状态，但没有执行完整 WCAG 人工审计，因此不宣称全量 WCAG 合规。

## 余留边界

1. `brandaudit` 是用于验证错误边界的错配种子账号，当前已可解释但不是可工作的产品角色；生产种子应删除或改成平台范围审计账号。
2. 角色首页体验配置与后端 RBAC 是不同层次的配置。当前用权限过滤和单测防止死链；后续只有在角色频繁变化时，才值得增加服务端下发的任务配置，避免为当前规模引入额外系统。
3. 本轮浏览器多角色矩阵为真实人工自动化回归，现有 CI Playwright 尚未逐项固化这些角色断言。若角色或门户信息架构再次调整，应把最小的首页标题、主动作和死链断言加入 CI。
