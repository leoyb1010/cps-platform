# 下一轮升级规划 v6

> 写在前面：本规划基于系统**真实现状**，不是愿望清单。
> 经过两轮对抗式安全自审，已修复 **19 个真实缺陷**（资金双花、提权链、越权泄漏、令牌盗用、
> 输入越界、错误码、ID 碰撞、分页不稳定……）。当前工程质量已是「**接近商用形态的全栈底座**」。
> 但「**真正商业可用**」仍卡在两类东西上：① 我能写但还没写的工程增量；② 代码替代不了的线下资质。
> 下面把①拆成可执行、可验证的批次，并把②再次诚实标注。

## 二、下一轮工程批次（我能做、可验证，按性价比排序）

### 批次 A — 可观测性与运维成熟度（性价比最高，~1 轮）
当前 metrics 是自研内存计数，日志是 pino，但缺「能在生产定位问题」的闭环。
1. **请求追踪 ID 贯穿**：每请求生成/透传 `X-Request-Id`，注入 pino + 错误响应 + 审计行，
   做到「一条用户报错 → 一行日志 → 一条审计」可串联。
2. **prom-client 替换自研 metrics**：标准直方图(请求延迟 P50/P95/P99) + 业务指标
   (资金动作计数、退款金额、审计失败计数)，对接 Grafana 看板。
3. **结构化健康检查升级**：`/ready` 增加「DB 连接池水位 + 最近一次审计写入成功时间」，
   让审计 fail-closed 的丢失可被探针发现。
4. **优雅停机**：`enableShutdownHooks` + 在途请求 drain，避免部署时截断资金事务。
- 验证：注入故障(DB 断连/审计失败)看探针/指标是否如实反映；负载脚本打 P95。

### 批次 B — 数据与一致性加固（~1 轮）
1. **切 PostgreSQL 实跑验证**：当前 PG schema 已 validate 但未实机迁移。
   用 docker-compose.pg.yml 起真 PG，跑全量 e2e（需 Docker 环境）。⏳ **待环境**
2. ~~**数据库唯一约束补全**：MerchantAccount.mid `@unique`~~ ✅ **已做(v7)**：
   重复 mid → P2002 → F3 映射 409。
3. **金额用整数分存储**（Float→Int 分）：🟡 **已研判，暂缓（v7）**。
   - 现状核查：所有金额运算的 `amt`/`share` 都已是整数（`Math.abs(order.amount)` + `Math.round`），
     应用层从不产生小于 1 元的金额，故 `0.1+0.2` 类浮点漂移**当前不可触发**。
   - Float **列类型**仍是潜在风险，但全量迁移需同步重写：seed 金额(×100)、所有 formatter(÷100)、
     share 口径——半途迁移比不迁移更危险。
   - **生产迁移步骤（待整体执行）**：① schema 金额列 Float→Int；② seed 全部 ×100；
     ③ 前端 format 层统一 ÷100 展示；④ 回归全部资金 e2e。建议与「切 PG」同一轮做。
4. **审计 before/after 全量 diff**：当前仅部分写操作带 before/after，补齐所有写操作。⏳ **待办**
5. ✅ **定时对账任务（已做 v7）**：`ReconciliationService` 核对各品牌退款↔冲账，
   差异写审计告警；`POST /reconciliation/run` 手动触发(super)，幂等只读。生产接 cron 即可。
- 验证：对账已实测(检查 5 品牌返回差异列表)；唯一约束实测 P2002→409。

### 批次 C — 安全纵深再加一层（~1 轮）
1. **CSRF 双提交令牌**：刷新令牌走 cookie，配套 CSRF token 防跨站请求。
2. **审计失败旁路落盘**：fail-closed 之外，best-effort 审计失败时写本地 append 文件，
   保证「即使 DB 审计表挂了也不丢线索」。
3. **敏感字段加密/脱敏**：商户号 mid、个人佣金等 PII 列加密存储 + 列表脱敏展示（《个保法》）。
4. **登录异常检测**：异地/异常频率登录 → 风控事件 + 可选二次验证钩子（不接真短信，留接口）。
5. **依赖漏洞扫描接 CI**：`npm audit` + Dependabot 工作流。
- 验证：CSRF 缺 token 被拒；脱敏后接口不返回明文；audit 文件旁路在 DB 故障时仍记录。

### 批次 D — 产品完备性（按需，~1-2 轮）
1. **业务页全量写操作接后端**：当前读已水合、写已镜像，但「新建品牌/号池」等创建类
   仍走本地乐观 + 镜像；改为创建后以服务端返回 id 为准（避免本地临时 id 漂移）。
2. **批量操作 + 导出**：结算批量核销、号池批量干预、审计/结算导出 Excel（带审计）。
3. **通知中心服务端化**：当前活动流是前端内存，改为服务端事件表 + 未读计数 + 订阅。
4. **配置中心结构化**：当前 config 是 KV，改为带 schema 校验的分组配置(阈值/通道/费率默认)。
- 验证：每个新写端点配 e2e；导出文件内容校验。

## 四、建议执行顺序与依赖

```
批次 A(可观测/运维) ──┐
                      ├─→ 批次 B(数据/一致性, 含 PG 实跑) ──→ 批次 D(产品完备)
批次 C(安全纵深) ─────┘
```

- **先 A**：没有可观测性，后面任何改动出问题都难定位——这是地基。
- **A 与 C 可并行**：互不依赖。
- **B 依赖环境**：PG 实跑需要一台装 Docker 的机器/云主机（我在当前沙箱无法验证容器）。
  其余 B 项（金额整数化、唯一约束、对账）可先在 SQLite 做。
- **D 最后**：产品完备性是锦上添花，价值低于前三者的「不出事」。

## 五、一句话结论

工程侧，下一轮最该做的是 **批次 A（可观测性/运维成熟度）**——它把系统从「能跑、正确」推向
「**生产可运维、出问题能定位**」，是通往真上线的下一块地基。

但请记住：**这些工程批次全部做完，系统也只是「工程满分」，不等于「商业可用」。**
商业可用的钥匙在第三节那些线下资质手里，那条线越早由你方启动越好。

我能做的，是把工程这条线一直推到顶；并在每一轮继续对自己的代码做对抗式审查，
而不是宣布「已经最优了」。
