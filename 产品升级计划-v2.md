## 0. 现状诊断（一句话）

当前产品是一套**高保真「可视化原型」**：展示层完整且精致（精密仪表风格、13 个页面、筛选/标签/图表/十字线/动效都可交互），但**业务闭环为零**——

- **15+ 个操作按钮全是装饰**（导出报表、本期结算预览、新增商户号、邀请品牌入驻、审核入驻申请、进入坐席台、分账通道配置、保存配置、领取追踪链接、规则配置…全部点了没反应）
- **0 个真实表单**（没有入驻表单、配置编辑、工单回复、退款操作、商户号录入）
- **只有 1 个详情页**（品牌详情）；代理 / 订单 / 工单 / 商户号 / 结算单 都没有详情页
- **"实时"数据是静态的**（订单流不滚动、计数不更新）
- **大部分指标没有锁定口径**（只有 8 个指标有定义气泡，其余靠自解释）
- **无全局能力**：搜索、时间范围、权限实装、通知中心、操作审计、空/加载/错误态、移动端、暗色、数据持久化全缺

**这一轮的目标**：把"能看"升级成"能用、能跑通闭环、口径清晰"，并铺好"走向真实生产"的轨道。

## 1. 三级成熟度模型（升级路径）

| 级别 | 名称 | 状态 | 本轮覆盖 |
|---|---|---|---|
| L1 | 可视化原型 | ✅ 已完成 | 展示 + 视觉系统 |
| **L2** | **可演示闭环** | 🎯 本轮主攻 | 详情页 + 表单 + 工作流 + 模拟数据驱动的真实交互（前端 mock service） |
| L3 | 可上线生产 | ⏭ 下一阶段 | 真实后端 API、清结算引擎、持牌分账对接、回传对接、权限/审计/多租户 |

> 关键判断：**先把 L2 做透**（用一个前端 mock 数据服务层模拟后端，让所有按钮/表单/流程真的能点、能改、能跑通状态流转），既能用于招商演示与内部验证，又能直接定义清楚 L3 的 API 契约。不要跳过 L2 直接接后端。

### 1.A · L2 能力深化（可演示闭环 —— 本轮地基）

> "可演示闭环"不是"点了有反应"那么浅。它的硬标准是：**每个操作真实改数据、状态真实流转、跨模块真实联动、刷新后状态还在**。分五块。

**① Mock 数据服务层（`lib/store.ts` + `lib/api.ts`）—— 本轮第一件该做的事**
- 能力：CRUD、状态机流转、**localStorage 持久化**（刷新不丢）、网络延迟模拟、**错误注入**（用于测错误态）、**seed / reset**（演示一键重置）、**事件总线 emit/subscribe**（驱动"实时"与跨模块联动）、乐观更新。
- 资源化（接口形态 = 未来 L3 API 契约蓝本）：`brands / plans / channels / merchantAccounts / agents / orders / subscriptions / trackLinks / settlements / ledger / withdrawals / invoices / tickets / riskRules / riskEvents / lists / reconciliations / attributionConfig / members / roles / auditLogs / configs`。

**② 状态机清单（必须实现，每个都要可流转）**
| 实体 | 状态 | 关键转移（触发 → 守卫 → 副作用） |
|---|---|---|
| 订单 | 待支付 / 已扣费 / 已退款 / 已拒付 | 退款（工单或规则 → 退款窗口校验 → 发"逆向冲账"事件） |
| 订阅 | 试用 / 续费中 / 已退订 / 已逾期 | 退订（用户/客服 → 停止续费计划 → 更新续费率） |
| 结算单 | 待结算 / 对账中 / 已结算 / 逆向冲账 / 账期冻结 | 发起（选周期 → 分润计算 → 冲账核对 → 冻结 → 二次确认） |
| 提现 | 待审 / 已审 / 打款中 / 已到账 / 驳回 | 审核（信用分 + 风险校验 → 通过/驳回） |
| 工单 | 待处理 / 处理中 / 已解决 / 仲裁中 | 退款→联动订单+结算冲账；升级→SLA 重算；超时→自动升级 |
| 商户号 | 健康 / 整改警告 / 暂停新签7 / 暂停新签21 / 暂停交易 | 阈值触发（自动）+ 人工干预；副作用：路由权重↓ + 反向收紧投放 |
| 品牌入驻 | 草稿 / 审核中 / 在投 / 暂停 / 下线 | 审核通过 → 生成专属号池 + 套餐上架选品市场 |
| 代理入驻 | 待审 / 在投 / 限流 / 冻结 / 黑名单 | 风控触发 → 降权/冻结；清退 → 主体/设备/收款账户三要素入黑名单 |

**③ 跨模块联动（L2 的灵魂 —— 这正是之前方案缺的"深度"）**
用事件总线把分散的模块串成"一条线"，例如：
- **退款链**：工单点「退款」 → 订单转「已退款」 → 生成「逆向冲账」 → 结算单冲减该代理已结/未结分润 → 代理「待结算」下降 → 代理「退款率/信用分」更新 → 若超阈值 → 代理「限流」 → 关联商户号投诉率↑ → 可能触发「整改警告」 → 首页 Risk Bar 变色 + 生成待办。
- **保号链**：某商户号投诉率逼近红线 → 自动「降低进单权重」 → 路由把流量切到健康号 → 触发「投放收紧」提示 → 首页风险条变黄 + 待办新增。
- **月结链**：发起月结 → 三方对账 → 差异挂起 → 进首页待办 → 人工核销 → 结算单「已结算」 → 平台净收入入账。
- **要做**：实现一个轻量 `eventBus`，把上述 3 条链路在前端真实跑通。**这是演示价值与定义后端契约的核心。**

**④ 演示剧本（seed scenarios，一键触发，用于招商/验收）**
- 剧本 A「保号危机」：号 M-XM-02 投诉率冲到 0.98% → 自动限流 → 升级投诉破 0.1% → 暂停新签 → 投放收紧 → Risk Bar 红。
- 剧本 B「升级投诉闭环」：监管投诉进入，SLA 18 分钟 → 客服退款 → 冲账 → 代理分润回收 → 工单关闭。
- 剧本 C「月结对账」：发起月结 → 芒果 TV 差异 ¥31,200 挂起 → 核销 → 完成。
- 剧本 D「代理刷量」：A-4410 空包单激增 → 风控命中 → 降权 → 信用分跌 → 限流。

**⑤ 交互完整度矩阵**：每页"按钮/表单/流程"逐项列出并标注 mock 实现状态（待办 / 进行 / 完成），作为 L2 的验收清单（替代现在"按钮全是装饰"的状态）。

### 2.1 首页（经营总览）重构 —— ★★ 本轮重点 ★★

> 当前首页是"每个格子塞一个好看的图表"，不是从**运营者每天要做的决策**倒推出来的。下面从业务逻辑重做。

#### 2.1.1 首页是谁在看、要回答什么（JTBD，按优先级）
平台运营者（你）打开首页，脑子里的问题**有严格优先级**：
1. **我的商户号今天安全吗？会不会被封？** —— 这是**存亡级**日常风险。号被封→进单中断→营收瞬间归零。（投诉率/升级投诉/72h完结/号池状态）
2. **现在有什么必须我今天处理的事？** —— 人在环：监管投诉、暂停新签、刷量、对账差异。
3. **生意健康吗、在赚钱吗？** —— 北极星 净LTV÷CAC、GMV、平台净收入、续费率。（慢变量，看趋势）
4. **资金安全吗、对得平吗、结算按时吗？** —— 待结算、对账差异、逆向冲账、风险准备金、二清敞口=0。
5. **上下游两侧健康吗？** —— 上游品牌供给、下游代理质量与刷量风险。

**结论：首屏第一眼必须先回答 1 和 2（风险+行动），而不是经营结果。** 当前首页恰恰把风险埋在第三行，把"看着热闹但没决策价值"的东西放在黄金位——这就是"指标不清楚为什么放在这里"的根因。

#### 2.1.2 当前首页逐区块诊断（为什么它该/不该在这里）
| 区块 | 当前位置 | 业务价值判定 | 处置 |
|---|---|---|---|
| 北极星仪表（净LTV÷CAC 2.34） | Hero 右 · 黄金位 | 战略慢变量；**与第一行 KPI 卡里的同一个 2.34 重复出现两次** | **去重**：KPI 行不再单列该数；仪表升级为承载"北极星 + 目标进度 + 净LTV/CAC 分解 + 趋势" |
| 资金清结算路径占比（直连51/持牌34/混合15） | Hero 右下 · 黄金位 | **结构性慢变量，日常零决策价值**（占比今天 51% 明天还是 51%，看了不触发任何动作）。占着首屏最贵的位置 | **下沉到「资金合规」页**。首屏这个位置改放"资金/结算健康"（待结算、对账差异、风险准备金覆盖、二清敞口） |
| 实时订单流（全部订单滚动） | Row D 左 | **Vanity 组件**：看单条正常订单滚过去，运营做不了任何决策（"活着的感觉"≠价值） | **改造为「异常订单流」**：只滚动 退款/拒付/大额/风控命中 的单——这些才需要关注与处置 |
| 品牌表现 | Row D 右 | 上游供给健康，**有价值**，但**三方市场首页只有上游没有下游** | 保留；**补对称的"代理（下游）质量汇总"** |
| 投诉率大盘 / 号池健康联动 | Row C（第三行） | **保号命脉**，价值最高的风险指标 | **上提到首屏**（风险优先于经营结果） |
| 今日待办 | Row C 右（第三行） | **人在环行动入口**，价值最高之一 | **上提，做成首屏「行动中心」** |
| 4 个经营 KPI（LTV/CAC、GMV、净收入、续费率） | 第一行 | 都是**经营结果/慢变量**；导致"首屏没有任何今天的风险信号" | 下移半屏到"经营健康"层；第一行让位给"风险状态 + 行动" |

#### 2.1.3 "上首页资格"准则（以后加任何指标都用它卡）
一个指标/区块要上首页，**必须满足至少一条**，否则下沉到二级页：
1. **关乎存亡**（保号红线、二清敞口、合规红线）
2. **是北极星或其直接驱动**（净LTV÷CAC、续费率）
3. **今天需要人工行动**（待办/告警）
4. **是规模/营收的最高层汇总**（GMV、平台净收入）
> 反例：资金路径结构占比、单条正常订单流、各品牌资质——都不满足，应下沉。

#### 2.1.4 重构后的首页信息架构（三层，按 JTBD 排序）

**层 1 · 风险与行动（首屏最显眼）——「今天安全吗？要我做什么？」**
- ★ **平台健康状态条（Risk Bar）**：一行红/黄/绿信号灯，聚合：号池（暂停新签 X / 熔断 Y）、综合投诉率（距红线还有多少）、升级投诉率、SLA 临期数、对账差异额、**二清敞口（必须=0）**、风险准备金覆盖率。**一眼判断有没有起火**。任一转红→整条变色+定位到源头。
- ★ **行动中心（今日待办，升级版）**：分级排序（监管投诉 > 暂停新签 > 刷量预警 > 对账差异），**每条可一键处理**（跳详情并直接打开操作抽屉），处理完即时消失/计数减少。

**层 2 · 北极星与经营健康——「生意健康吗？」**
- ★ **北极星仪表（唯一一处）**：净LTV÷CAC + 目标进度（距 3.0）+ 净LTV/CAC 分解。
- ★ **经营 KPI**：GMV、平台净收入、续费率（各带 较目标 / 同环比 / 迷你趋势）。
- ☆ **GMV 与净收入 12 月趋势**（慢变量放这层，配十字线）。

**层 3 · 供需两侧 + 风险详情——「上下游和风险面怎么样？」**
- ★ **号池健康联动**（命脉详情，带预警水位）+ **投诉率/升级/72h完结 趋势**。
- ★ **上游：品牌表现** ‖ **下游：代理质量汇总**（活跃数、高风险数、待结算额、刷量预警数、信用分分布）——**补齐三方对称**。
- ☆ **异常订单流**（改造后）+ **渠道/资金路径分布**（下沉版，放这里而非黄金位）。

> 每个区块在产品里都要能回答四件事（建议做成 hover 说明）：**为什么在这里（对应哪个 JTBD）· 驱动什么决策/行动 · 点击去哪 · 异常时怎样**。

## 3. Part B — 指标与口径字典（解决"定义不清晰"）

> 这是本轮**最高价值**的产物之一。每个指标必须有：定义、公式、口径范围、边界规则、负责人。下表是需要锁定的全集（建议在产品内做一个「口径说明」页 + 每个指标旁挂气泡）。

### 3.1 北极星与营收
| 指标 | 精确定义 / 公式 | 口径要点 / 边界 |
|---|---|---|
| 净 LTV | 单用户生命周期**扣除退款/拒付后**的累计净付费 | 需定义统计窗口（如 12 个月）、是否含三方权益成本；按 cohort 还是滚动 |
| CAC | 获客成本 = 投放消耗 ÷ 新增付费用户数 | 分母是"首单付费"还是"激活订阅"；是否含平台/代理双口径 |
| 净 LTV ÷ CAC | 北极星；健康线需明确（建议 >2 可接受、>3 健康） | 当前仪表"健康(>2.0)"与"目标 3.0"并存，需统一表述 |
| GMV（本月） | 当月**全部品牌**扣款成交额合计（首单+续费） | = 各品牌 gmvMtd 之和；含/不含退款冲减需声明 |
| GMV（年累计） | 当年各月 GMV 之和 | 与月度口径一致 |
| 平台净收入 | 可分配池 − 代理分润 | 可分配池 = GMV × 品牌费率；是否扣通道费/税费 |
| 续费率（连续包月） | 到期应续中实际续费占比 | 按"次月续费"还是"周期续费"；分母如何定义 |
| 续费 ÷ 首单 | 续费笔数 ÷ 首单笔数（已标注） | 同周期口径 |

### 3.2 风控与合规（支付平台口径，已部分锁定）
| 指标 | 精确定义 / 公式 | 红线 / 阈值 |
|---|---|---|
| 投诉率 | 近 7 天扣款交易投诉数 ÷ 近 7 天扣款交易数（**7 天累计**） | < 1%；平台内部预警 0.6% |
| 升级投诉率 | 升级到平台/监管的投诉占比 | < 0.1%；0.05–0.1% 整改警告 |
| 72h 投诉完结率 | 72 小时内完结的投诉占比 | ≥ 95%；90–95% 整改警告 |
| 拒付率 / 退款率 | 拒付（chargeback）/ 退款 笔数或金额占比 | 需声明按笔还是按额 |
| 商户号状态机 | 健康→整改警告→暂停新签(7天)→暂停新签(21天)→暂停交易 | 各档触发条件需逐条写死规则表 |
| 风险准备金 | 按品牌比例对分润账期冻结的资金 | 冻结比例、释放条件（对齐渠道可退周期 30–180 天） |
| 账期冻结 | 覆盖退款拒付窗口的冻结额（已标注） | 释放触发器需定义 |
| 信用分（代理） | 投诉率/退款拒付/刷量嫌疑/合规违规/结算纠纷 加权 | **权重与计算公式当前只在文案里，需落为可配置规则** |

### 3.3 投放与归因
| 指标 | 精确定义 | 口径要点 |
|---|---|---|
| ROI | 订单收入 ÷ 投放消耗 | 首单 ROI vs 含续费 ROI 要分开 |
| CPA | 投放消耗 ÷ 转化数 | 转化 = 首单付费 |
| 归因覆盖率 | 可归因转化 ÷ 总转化 | 去重、点击劫持处理后 |
| 归因窗口 | 点击/曝光到转化的有效归因时长 | 首单与续费分别定义 |
| LTV 预测（T+0） | 首单期即预测的净 LTV | 模型、误差范围（当前 ±6.8% 写死）、输入特征 |
| 同期群留存 | 开月=100%，逐月留存衰减 | 已用「同期群/开月/N月」表述 |

### 3.4 枚举与字典（需统一并集中维护）
- **订单类型**：首单 / 续费 / 退款 / 拒付
- **资金路径**：直连 / 持牌分账 / 混合（定义与适用品牌规则）
- **商户号状态**：健康 / 整改警告 / 暂停新签·7天 / 暂停新签·21天 / 暂停交易
- **代理状态**：正常 / 限流 / 冻结结算 / 黑名单
- **工单来源**：平台内 / 支付渠道 / 12315 / 黑猫 / 应用商店
- **工单等级**：普通 / 升级投诉 / 监管投诉
- **工单状态**：待处理 / 处理中 / 已解决 / 仲裁中
- **结算状态**：待结算 / 已结算 / 对账中 / 逆向冲账 / 账期冻结
- **开票方式**：灵活用工 / 企业开票 / 个体户

> 建议：把上述全部枚举与口径**集中到一份配置（前端 `lib/dict.ts` / 后端字典表）**，页面只引用，避免散落、口径漂移。当前枚举已在 `data.ts`，但口径文字分散在各页 —— 收口为单一可信源。

## 5. Part D — L3 能力深化（走向真实生产）

> 以 L2 的 mock service 接口为契约蓝本逐资源落地。分六块，每块给到可落地的颗粒度。

### 5.1 API 契约（按资源）
统一约定：所有写操作带**幂等键**；资金类操作**二次确认 + 强制审计**；列表支持过滤/排序/cursor 分页；统一错误码；字段命名与口径以 Part B 字典为准。

| 资源 | 关键端点（除标准 CRUD 外） |
|---|---|
| brands / plans / channels | 入驻提交、审核(通过/驳回)、上下线、回传配置、套餐定价历史 |
| merchantAccounts | 录入、状态干预(限流/暂停/恢复/熔断)、阈值配置、路由权重、路由日志 |
| agents | 入驻审核、信用分调整、限流/冻结/清退、保证金、开票资料 |
| orders / subscriptions | 详情、退款、退订 → 触发冲账；多维查询、导出 |
| trackLinks | 生成(代理×品牌×套餐×渠道×素材)、二维码、归因绑定 |
| settlements / ledger | 发起结算、对账、冻结/释放、结算单明细、资金台账(可审计导出) |
| withdrawals / invoices | 提现发起/审核/打款回执、开票申请/进度 |
| tickets | 处理(退款/退订/升级/转派/关闭/补发)、SLA 配置、多源接入 |
| riskRules / riskEvents / lists | 规则增删改、事件复核、黑/白/灰名单 |
| reconciliations | 三方对账拉取、差异挂起、核销 |
| attributionConfig | 归因窗口、去重、首单/续费归因、回传对接状态 |
| members / roles / auditLogs / configs / webhooks | RBAC、审计查询、配置持久化、Webhook |

### 5.2 领域引擎（后端核心，给职责 + 不变量）
- **清结算引擎**：分润分层（品牌费率→平台费→代理分润）、**逆向冲账**（可回收已结分润）、账期冻结/释放、税费、生成结算单与台账。*不变量：每笔钱来源可追溯、可冲正、对账恒平、操作幂等。*
- **对账引擎**：平台采集 vs 品牌回传 vs 支付/分账，逐笔比对、差异分类、挂起队列、人工核销留痕。
- **风控引擎**：实时防作弊（同设备/IP、异常时差、空包/秒退、归因劫持、异常退款集中）、规则引擎（条件→动作）、名单库、信用分计算（权重可配，见 Part B）。
- **路由引擎**：号池健康加权进单、阈值熔断、容灾切换、冷却恢复；**反向输出投放收紧指令**。
- **归因引擎**：统一 ID、多平台转化回传、首单/续费归因、去重/防劫持、**T+0 LTV 预测**。
- **状态机引擎**：统一驱动各实体流转，事件溯源、幂等、审计。

### 5.3 集成适配器（可插拔 Adapter）
- **持牌分账**：连连 / 汇付 / 微信官方分账 / 支付宝分账 / 银行存管 —— 分账指令下发、对账回执、双通道容灾。**只下发指令、不碰资金本体，规避二清**（v1 §3.2 路径 B）。先接 1 家 MVP + 1 家容灾。
- **支付签约**：微信 / 支付宝 连续包月 签约 + 扣费 + 退订 能力对接。
- **转化回传**：巨量 / 快手 / 广点通 / 百度 / 支付宝 转化 API（回传"签约/支付成功"优化投放模型）。
- **开票**：灵活用工 / 众包平台（个人佣金合规开票，v1 §6.3），三流一致。
- **通知**：短信 / 企业微信 / 钉钉（风险告警、SLA 临期、结算完成）。

### 5.4 数据平台
- **指标计算落地**（口径以 Part B 单一可信源为准）。
- **实时流**：投诉率 7 天滚动窗、SLA 计时、号池健康——需要流式/定时计算（Flink/定时任务）。
- 数仓分层、归因计算管道、LTV 模型训练/推理。

### 5.5 非功能（生产必备，少一个都不能上线）
- **多租户隔离** + **RBAC 三级**（路由级 / 操作级 / **数据级**：品牌只见自己、代理只见自己）。
- **全量审计**（资金 / 权限 / 配置 变更全留痕，可取证）。
- **加密与脱敏**（手机号、支付信息；遵《个人信息保护法》）。
- **幂等与对账**（所有资金操作幂等，每日三方对账）。
- 限流、监控告警、灾备、SLA。

### 5.6 迁移
- 把现有**有道业务作为首个租户**迁入（v1 §11 Phase 0）：数据迁移、配置化校验、灰度放量、新老双跑对账后切换。

## 8. 优先级与分期路线图

### Phase 1（L2 地基 + 首页重构，1–2 周）— 让产品"能用"且首屏对
1. ★ Mock 数据服务层（`lib/store.ts` + `lib/api.ts` + localStorage 持久化 + 状态流转）+ **事件总线 `eventBus`**（联动地基）
2. ★ 设计系统补全：Modal/Drawer/Toast/Form/Pagination/EmptyState/Timeline/Steps
3. ★ **首页重构（重点）**：Risk Bar 风险状态条 + 行动中心(可一键处理) + 版面重排(风险上提/经营下移/北极星去重) + 代理侧汇总 + 异常订单流改造 + 资金路径下沉
4. ★ 全局时间范围 + 表格能力（分页/排序/导出）+ 空/加载/错误态
5. ★ 口径字典收口（`lib/dict.ts`）+ 产品内「口径说明」页 + 全指标气泡

### Phase 2（核心闭环，2–4 周）— 让钱与风险跑通
5. ★ 清结算：发起流 + 对账中心 + 结算单详情 + 代理提现流
6. ★ 商户号：录入 + 详情 + 状态机手动干预 + 阈值/权重配置
7. ★ 投诉工单：处理台 + 详情 + 一键退款/退订(联动冲账) + SLA 配置
8. ★ 订单：详情 + 退款/退订流 + 多维筛选/搜索/分页

### Phase 3（招商与增长闭环，2–3 周）
9. ★ 品牌入驻向导 + 审核流 + 配置编辑；选品市场领链接 + 素材库
10. ★ 代理：详情 + 入驻审核 + 信用分实装 + 黑名单库 + 风控操作
11. ★ 风控：规则引擎配置 + 事件详情 + 复核流 + 名单管理
12. ☆ 数据归因：归因配置 + 维度下钻；资金合规：分账通道配置 + 路径切换

### Phase 4（平台化与生产准备，3–4 周）
13. ★ RBAC 实装 + 操作审计 + 通知中心
14. ☆ 响应式/移动端 + 暗色模式
15. ⏭ L3：API 契约冻结 → 后端/引擎/分账/回传对接（另立项目）

## 9. 风险与依赖

| # | 风险/依赖 | 说明 |
|---|---|---|
| 1 | 口径不统一会污染一切 | 指标字典(Part B)必须先锁，否则前端/后端/对账各算各的 |
| 2 | 跳过 L2 直接接后端 | 会导致 API 契约反复返工；先用 mock service 把交互与流程定死 |
| 3 | 资金/合规操作不可逆 | 结算发起、提现、冲账、商户号熔断必须二次确认 + 审计 + 权限 |
| 4 | 移动端是原始诉求 | 设计阶段就要求响应式，越晚补成本越高 |
| 5 | 真实时数据 | 订单流/风控/SLA 计时最终需 WS/SSE，L2 先用轮询 mock |
